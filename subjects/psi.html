Počítačové sítě

1. Referenční model OSI/ISO – vrstvy, jejich funkce, smysl zavedení

Počítačové sítě jsou komplexní systémy. Aby různí výrobci mohli navzájem komunikovat, potřebovali společný jazyk — standard, který říká, kdo se stará o co. Tím je referenční model OSI (Open System Interconnection), vydaný organizací ISO jako norma IS 7498 v roce 1984.

Intuice: Představ si posílání dopisu. Nestaráš se o to, jak pošta třídi zásilky, jakou trasou letí letadlo nebo jak řidič najde tvou ulici. Každá "vrstva" systému dělá svou část a od ostatních abstrahuje. OSI přesně takto vrství síťovou komunikaci.
Definice: Referenční model OSI je sedmivrstvá architektura popisující, jak mají být funkce síťové komunikace rozděleny do logických vrstev. Každá vrstva poskytuje služby vrstvě nad ní a využívá služby vrstvy pod ní. Komunikace probíhá vždy mezi stejnými vrstvami na obou koncích (peer-to-peer).

Sedm vrstev (od nejnižší)

Proč vrstvit?

Princip vrstvení přináší modulárnost: každá vrstva lze nahradit jinou implementací bez zásahu do ostatních. WiFi a Ethernet mohou coexistovat — obě přenáší IP pakety, přestože se fyzicky liší. Výrobce routeru se nemusí starat o kódování bitů, výrobce síťové karty nemusí implementovat HTTP.

Příklad: Když otevřeš webovou stránku: prohlížeč (vrstva 7) sestaví HTTP požadavek → TLS (vrstva 6) ho zašifruje → TCP (vrstva 4) ho rozdělí na segmenty → IP (vrstva 3) přidá adresy → Ethernet (vrstva 2) zabalí do rámce → fyzická vrstva pošle bity po kabelu. Na serveru se vše rozbalí v opačném pořadí.
Shrnutí:
  • OSI má 7 vrstev; každá řeší jednu část problému síťové komunikace.
  • Fyzická (bity) → Linková (rámce, MAC) → Síťová (IP) → Transportní (TCP/UDP) → Relační/Prez./Aplik.
  • Smysl: modulárnost, interoperabilita výrobců, oddělení zodpovědností.
  • V praxi TCP/IP model slučuje vrstvy 5–7 do jedné aplikační.

2. Síťová vrstva – IP adresa, struktura, maska sítě, podsítě

Linková vrstva ví, jak doručit rámec na lokální síti. Ale jak poslat data přes celý internet — přes stovky různých sítí? Potřebujeme globální logické adresování a mechanismus, který rozhodne, kudy data pošleme. To zajišťuje síťová vrstva s protokolem IP.

Definice (IPv4 adresa): 32bitové číslo jednoznačně identifikující síťové rozhraní v internetu. Zapisuje se jako čtyři decimální oktetty oddělené tečkami (dotted-decimal notation), např. 192.168.1.10. IP adresa se logicky skládá ze dvou částí: síťové části (identifikuje síť) a hostitelské části (identifikuje zařízení v síti).

Maska sítě (subnet mask)

Maska sítě je 32bitové číslo, kde jedničky označují síťovou část adresy a nuly hostitelskou část. Zapisuje se buď jako 255.255.255.0 nebo v CIDR notaci jako /24 (počet jedničkových bitů).

Příklad výpočtu: IP adresa 192.168.1.10, maska /24 (= 255.255.255.0):
  • Adresa sítě (AND s maskou): 192.168.1.0
  • Broadcast: 192.168.1.255 (všechny bity hostitele = 1)
  • Rozsah hostů: 192.168.1.1 – 192.168.1.254 (254 zařízení)
Obecně: pro masku /n je počet hostů $2^{32-n} - 2$.

Podsítě (subnetting)

Velkou síť lze rozdělit na menší podsítě — prodloužíme masku a část hostitelských bitů "ukradneme" pro identifikaci podsítě. Důvody: bezpečnostní izolace, snížení broadcast domény, efektivnější využití adresního prostoru.

Příklad: Ze sítě 192.168.1.0/24 vytvoříme 4 podsítě (/26):
  • 192.168.1.0/26 — hosté .1–.62
  • 192.168.1.64/26 — hosté .65–.126
  • 192.168.1.128/26 — hosté .129–.190
  • 192.168.1.192/26 — hosté .193–.254
Pozor: Adresa sítě (všechny hostitelské bity 0) a broadcast (všechny hostitelské bity 1) nejsou použitelné pro přidělení hostům — každá podsíť "ztrácí" 2 adresy.
Shrnutí:
  • IPv4 adresa = 32 bitů = síťová část + hostitelská část.
  • Maska sítě (AND s IP) určí adresu sítě; CIDR notace /n říká, kolik bitů je síťových.
  • Podsítě = prodloužení masky = logické rozdělení jedné velké sítě na menší.
  • Každá podsíť má 2 nepoužitelné adresy: adresa sítě a broadcast.

3. Směrování (routing) – principy, routovací tabulka, default gateway

IP datagram putuje sítí od routeru k routeru. Každý router rozhoduje: "Kam pošlu tento paket dál?" Toto rozhodnutí se dělá na základě routovací tabulky — mapy cest, kterou si router udržuje.

Definice: Směrování (routing) je proces výběru cesty, po které putuje IP paket od zdroje k cíli přes sítě propojené routery. Router při přijetí paketu porovná cílovou IP adresu se záznamy v routovací tabulce a pošle paket na příslušné rozhraní (next-hop).

Routovací tabulka

Každý řádek tabulky obsahuje: cílovou síť, masku, next-hop IP (nebo rozhraní) a metriku (cenu cesty). Router hledá nejdelší shodu prefixu (longest prefix match) — specifičtější záznam má přednost.

Příklad routovací tabulky:
Cílová síťMaskaNext-hopRozhraní
192.168.1.0/24přímo připojenáeth0
10.0.0.0/8192.168.1.1eth0
0.0.0.0/0192.168.1.254eth0
Paket pro 10.5.3.2 → druhý řádek (10.0.0.0/8). Paket pro cokoliv jiného → třetí řádek (default route).

Default gateway

Koncové zařízení nemá komplexní routovací tabulku — zná jen svou síť a default gateway (výchozí bránu). Pokud cílová adresa není ve stejné síti (AND s maskou dá jinou síťovou adresu), paket se pošle na default gateway, která se postará o zbytek.

Statické vs. dynamické směrování

Záznamy v tabulce mohou být statické (ručně nakonfigurované administrátorem) nebo dynamické (automaticky aktualizované směrovacími protokoly jako RIP, OSPF, BGP). Dynamické protokoly sdílejí informace o sítích mezi routery a adaptují se na výpadky.

Shrnutí:
  • Router vybírá next-hop na základě longest prefix match v routovací tabulce.
  • Default route (0.0.0.0/0) = záchytná síť pro pakety bez specifičtějšího záznamu.
  • Default gateway = IP routeru, na který posílají pakety koncová zařízení v LAN.
  • Dynamické protokoly (RIP, OSPF, BGP) automaticky udržují tabulky konzistentní.

4. TCP/IP model – vztah k OSI, rozdíly

TCP/IP je praktická architektura, která vznikla dříve než OSI (ARPANET, 70. léta) a je základem dnešního internetu. OSI je teoretický referenční model; TCP/IP je to, co reálně běží na sítích.

Klíčové rozdíly

  • Počet vrstev: OSI má 7, TCP/IP má 4 (slučuje relační, prezentační a aplikační do jedné).
  • Vznik: OSI vznikl jako standard (top-down), TCP/IP empiricky z praxe ARPANETU (bottom-up).
  • Přísnost: OSI přesně definuje rozhraní mezi vrstvami; TCP/IP je volnější — protokoly mohou překrývat funkce.
  • Využití: OSI se používá jako pedagogický nástroj; TCP/IP je reálná implementace internetu.
Intuice: OSI je jako ideální šablona architekta. TCP/IP je jako dům, který byl postaven pragmaticky — funguje skvěle, ale ne vždy dodržuje šablonu do písmene. Dnes se OSI a TCP/IP vzájemně doplňují: TCP/IP popisuje co, OSI vysvětluje proč.
Shrnutí:
  • TCP/IP má 4 vrstvy: síťového přístupu, internetová, transportní, aplikační.
  • OSI má 7 vrstev — vrstvy 5, 6, 7 jsou v TCP/IP sloučeny do aplikační.
  • TCP/IP je praxe internetu; OSI je referenční model pro pochopení a srovnání.

5. Přístup ke sdílenému médiu – CSMA/CD, CSMA/CA

Pokud více zařízení sdílí jedno přenosové médium (např. koaxiální kabel nebo vzduch u WiFi), nastává problém: co se stane, když vysílají dvě zařízení najednou? Potřebujeme protokol pro koordinaci přístupu k médiu (Medium Access Control).

Definice (CSMA): Carrier Sense Multiple Access — každá stanice před vysíláním naslouchá (carrier sense), zda je médium volné. Pokud je obsazené, čeká. Přesto mohou nastat kolize (dvě stanice začnou vysílat téměř současně).

CSMA/CD (Collision Detection) — Ethernet

Stanice vysílá a zároveň naslouchá médiu. Pokud detekuje kolizi (zachytí jiný signál překrývající její vysílání), okamžitě zastaví vysílání, vyšle JAM signál (32 bitů) upozorňující ostatní, a pak čeká náhodnou dobu (exponential backoff) před opakováním.

Příklad CSMA/CD: Stanice A začne vysílat. Stanice B naslouchá, médium se jeví volné (signál od A ještě nedorazil — propagační zpoždění), B také začne vysílat. Obě detekují kolizi, vyšlou JAM, zastaví. Každá čeká náhodný násobek slot time (512 bitů na Ethernetu) před opakováním.

CSMA/CA (Collision Avoidance) — WiFi (802.11)

V bezdrátových sítích nelze detekovat kolize (zařízení nemůže zároveň vysílat a přijímat na stejné frekvenci). Proto se kolizím vyhýbáme předem: stanice čeká DIFS interval, pak čeká náhodný backoff (countdown), a teprve pak vysílá. Příjemce musí potvrdit ACK. Pokud ACK nedorazí, předpokládá se kolize a proces se opakuje.

CSMA/CDCSMA/CA
ProstředíDrátový EthernetBezdrátové sítě (WiFi)
PrincipDetekce kolize za běhuVyhýbání se kolizím předem

Pozor: CSMA/CD je dnes na Ethernetových sítích prakticky zaniklé — moderní switched Ethernet vytváří point-to-point spoje (full duplex), kde ke kolizím nedochází vůbec. CSMA/CD je relevantní jen pro historické (half-duplex) Ethernet nebo huby.
Shrnutí:
  • CSMA = naslouchej před vysíláním; CD = detekce kolize za běhu (Ethernet); CA = vyhýbání se kolizím (WiFi).
  • CSMA/CD: kolize detekována → JAM → exponential backoff → retransmit.
  • CSMA/CA: backoff před vysíláním + ACK potvrzení příjmu.
  • Switched Ethernet (full duplex) kolize eliminuje úplně.

6. Spojová vrstva – rámec (frame), struktura, MAC adresace

Spojová vrstva (vrstva 2) přebírá pakety od IP a balí je do rámců — jednotek přenosu na lokální síti. Stará se o adresování v rámci LAN pomocí MAC adres a o detekci chyb přenosu.

Definice (MAC adresa): 48bitová (6 bajtů) fyzická adresa síťové karty, přidělená výrobcem a globálně unikátní. Zapisuje se jako šestice hexadecimálních dvojic, např. AA:BB:CC:DD:EE:FF. První 3 bajty = OUI (identifikátor výrobce), poslední 3 = sériové číslo.

Intuice: IP adresa je jako PSČ a název ulice — globální adresování. MAC adresa je jako jméno na poštovní schránce — lokální identifikace. Při přenosu přes internet se IP adresa nemění, ale MAC adresa se mění na každém skoku (hop) mezi routery.
Příklad: Tvůj PC (MAC: AA:BB) posílá paket routeru (MAC: CC:DD). Ethernet rámec: Src=AA:BB, Dst=CC:DD. Router rámec rozbalí, podívá se na IP, sestaví nový rámec s Src=CC:DD a Dst dalšího hoppu. IP adresy se přitom nemění.
Shrnutí:
  • Spojová vrstva zabalí IP paket do rámce s MAC adresami pro lokální doručení.
  • MAC adresa = 48 bitů, přidělena výrobcem, globálně unikátní.
  • Ethernet rámec: preambule | cíl MAC | zdroj MAC | EtherType | payload | FCS.
  • FCS (CRC) detekuje chyby; poškozené rámce se zahazují (bez retransmisí — to je úkol TCP).

7. Přepínač (switch) – princip, přepínací tabulka, flooding

Hub posílal každý rámec na všechny porty — zbytečné a bezpečnostně nevhodné. Switch se naučí, která zařízení jsou na kterém portu, a posílá rámce cíleně.

Definice: Přepínač (switch) je aktivní síťové zařízení pracující na vrstvě 2 (linkové). Udržuje CAM tabulku (Content Addressable Memory, také MAC tabulka nebo přepínací tabulka) mapující MAC adresy na porty, a na základě ní přeposílá rámce pouze na příslušný port.

Jak se switch učí?

  1. Přijme rámec na portu P od stanice se zdrojovou MAC A.
  2. Zapíše do CAM tabulky: MAC A → port P (s TTL, typicky 5 minut).
  3. Podívá se na cílovou MAC B v tabulce.
  4. Pokud B nalezl → pošle rámec pouze na port B.
  5. Pokud B nenalezl → flooding: pošle rámec na všechny porty kromě příchozího.
Příklad flooding: Switch se právě restartoval, CAM tabulka je prázdná. Stanice A pošle rámec stanici B. Switch nezná MAC B → flood na všechny porty. B odpoví → switch se dozví port B a zapíše. Příště rámec pro B pošle cíleně.
Intuice: Switch funguje jako chytrá recepce — pamatuje si, v které místnosti (portu) sedí kdo (MAC). Zpočátku rozeslat zprávu všem, ale po první odpovědi si zapamatuje. Hub je jako hlasité hlášení do celého patra bez ohledu na adresáta.
Pozor: Broadcast rámce (cílová MAC FF:FF:FF:FF:FF:FF) se vždy floodují na všechny porty — to nelze obejít bez VLAN. Velká broadcast doména = zbytečná zátěž sítě.
Shrnutí:
  • Switch udržuje CAM tabulku: MAC → port; učí se ze zdrojových adres příchozích rámců.
  • Flooding = poslání na všechny porty, pokud cílová MAC není v tabulce.
  • Broadcast se vždy flooduje; VLAN omezuje broadcast doménu.
  • Na rozdíl od hubu switch nevytváří kolizní doménu — každý port je vlastní doménou.

8. DHCP – dynamické přidělení IP adresy

Každé zařízení v síti potřebuje IP adresu, masku, bránu a DNS server. Ruční konfigurace tisíců zařízení je nepraktická. DHCP (Dynamic Host Configuration Protocol) tento proces automatizuje.

Definice: DHCP je aplikační protokol (UDP, port 67/server, 68/klient), který umožňuje automatické přidělení IP adresy a dalších síťových parametrů klientům při připojení do sítě.

Průběh přidělení adresy (DORA)

  1. Discover: Klient (bez IP) vyšle broadcast: "Hledám DHCP server!" (Src: 0.0.0.0, Dst: 255.255.255.255)
  2. Offer: DHCP server odpoví: "Nabízím ti adresu X.X.X.X na Y sekund."
  3. Request: Klient (broadcast) říká: "Přijímám nabídku od serveru Z, chci adresu X.X.X.X."
  4. Acknowledge: Server potvrdí: "Adresa je tvoje na Y sekund (lease time)."
Co DHCP přiděluje: IP adresu, masku podsítě, default gateway, adresu DNS serveru, lease time (dobu platnosti), případně NTP server, domény apod.
Intuice: DHCP je jako hotelová recepce — přijdeš, řekneš, že potřebuješ pokoj, a recepce ti přidělí klíč od volného pokoje (IP adresu) na dobu tvého pobytu (lease time). Po odjezdu klíč vrátíš a pokoj dostane někdo jiný.
Pozor: Discover a Request jsou broadcast — DHCP server musí být ve stejné L2 síti nebo musí existovat DHCP relay agent (router přeposílá DHCP zprávy přes podsítě).
Shrnutí:
  • DHCP automatizuje přidělování IP adres, masek, brány a DNS serverů.
  • Proces DORA: Discover → Offer → Request → Acknowledge.
  • Používá UDP; Discover a Request jsou broadcast (před přidělením IP).
  • Lease time = délka platnosti přidělené adresy; poté se musí obnovit.

9. Transportní vrstva – TCP, navázání a ukončení spojení

IP doručuje pakety best-effort — bez záruky pořadí nebo doručení. Aplikace jako web nebo e-mail ale potřebují spolehlivý přenos. TCP (Transmission Control Protocol) buduje spolehlivý datový tok nad nespolehlivým IP.

Definice: TCP je spojově orientovaný, spolehlivý, plně duplexní transportní protokol. Garantuje doručení dat ve správném pořadí bez ztrát a duplikát. Identifikuje komunikující procesy pomocí čísel portů.

Three-way handshake (navázání spojení)

Příklad:
  1. SYN: Klient → Server. "Chci se spojit, moje počáteční pořadové číslo (ISN) je X."
  2. SYN-ACK: Server → Klient. "Přijato (ACK = X+1), moje ISN je Y."
  3. ACK: Klient → Server. "Přijato (ACK = Y+1)." Spojení je navázáno.
Proč tři kroky? Potřebujeme ověřit, že obě strany mohou vysílat i přijímat — to vyžaduje výměnu a potvrzení obou počátečních pořadových čísel.

Ukončení spojení (four-way handshake)

  1. FIN: Jedna strana (A) oznamuje konec vysílání.
  2. ACK: Druhá strana (B) potvrzuje.
  3. FIN: B oznamuje, že i ona skončila.
  4. ACK: A potvrzuje — spojení ukončeno.

Čtyři kroky, protože ukončení každého směru je nezávislé (half-close). A může přestat posílat, ale B ještě může data posílat.

Klíčové mechanismy TCP

  • Sekvenční čísla a ACK: každý bajt má číslo; příjemce potvrzuje přijatá data.
  • Retransmission: pokud ACK nepřijde v časovém limitu, data se pošlou znovu.
  • Flow control (okno): příjemce říká, kolik dat může přijmout (receive window).
  • Congestion control: TCP zpomaluje při přetížení sítě (slow start, AIMD).
Shrnutí:
  • TCP = spolehlivý, spojově orientovaný protokol na vrstvě 4.
  • Three-way handshake (SYN, SYN-ACK, ACK) pro navázání — ověří obousměrnou komunikaci.
  • Ukončení: four-way (FIN, ACK, FIN, ACK) — každý směr zavírán nezávisle.
  • Spolehlivost zajišťuje sekvenčními čísly, ACK a retransmisemi.

10. Bezpečnost sítí – VPN, filtrování MAC/IP adres

Veřejné sítě jsou nebezpečné — data mohou být odposlechnuta, modifikována nebo falšována. Bezpečnost sítí se zaměřuje na důvěrnost (confidentiality), integritu (integrity) a dostupnost (availability) — tzv. triáda CIA.

VPN (Virtual Private Network)

VPN vytváří šifrovaný "tunel" přes veřejnou síť, takže komunikace vypadá jako by probíhala na soukromé síti. Používá se pro vzdálený přístup zaměstnanců, propojení poboček nebo anonymizaci provozu.

IPSec VPN: Šifruje provoz na vrstvě 3. Vhodné pro site-to-site VPN (propojení poboček). Komplexnější konfigurace, ale lepší výkon pro velké objemy dat.
SSL/TLS VPN: Šifruje provoz na vrstvě 4–7. Funguje přes HTTPS (port 443) — prochází většinou firewallů. Vhodné pro remote-access VPN (přístup z internetu). Stačí webový prohlížeč nebo jednoduchý klient.

Filtrování MAC adres

Switch nebo WiFi access point může mít whitelist povolených MAC adres. Nepovoleným zařízením je přístup odepřen. Problém: MAC adresy lze snadno podvrhnout (MAC spoofing) — jde o slabou obranu, vhodnou jen jako doplněk.

Filtrování IP adres (ACL)

Router nebo firewall obsahuje Access Control Lists (ACL) — pravidla povolující nebo zakazující provoz na základě zdrojové/cílové IP adresy, portu a protokolu. ACL jsou základem paketových firewallů.

Pozor: MAC filtrování i IP filtrování jsou reaktivní opatření — útočník, který odposlouchává síť, může legitímní adresy zjistit a podvrhnout. Spoléhat pouze na tyto mechanismy bez šifrování je nedostatečné.
Shrnutí:
  • VPN = šifrovaný tunel přes veřejnou síť; IPSec (vrstva 3) nebo SSL/TLS (vrstva 4–7).
  • MAC filtrování = whitelist na L2; slabé (spoofing), vhodné jen jako doplněk.
  • ACL (firewally) filtrují provoz podle IP adres a portů na vrstvě 3–4.
  • Bezpečnost = kombinace: šifrování + autentizace + filtrování + monitoring.

Fyzická vrstva se stará o skutečný přenos bitů — jako elektromagnetický signál po kabelu, světelný pulz v optickém vlákně nebo rádiová vlna vzduchem. Volba média zásadně ovlivňuje rychlost, dosah a odolnost sítě.

  • Bezdrátové médium: sdílené, rušení, ale flexibilní nasazení.
  • Klíčové parametry: šířka pásma, útlum, přeslechy, latency.

12. Fyzická vrstva – realizace kanálu, kódování binárních dat

Fyzická vrstva musí zakódovat bity (0 a 1) do fyzikálního signálu, který přežije přenos po médiu. Různá kódovací schémata mají různé vlastnosti — synchronizace, šířka pásma, odolnost vůči chybám.

NRZ (Non-Return to Zero)

Jednoduchý princip: 1 = kladné napětí, 0 = záporné napětí. Signál se nevrací na nulu mezi bity. Problém: dlouhý řetěz stejných bitů → žádná změna signálu → ztráta synchronizace hodin.

Manchesterské kódování

Každý bit je zakódován jako přechod uprostřed bitového intervalu: 1 = přechod z nízké na vysokou úroveň, 0 = přechod z vysoké na nízkou. Přechod vždy existuje → samosynchornizace (receiver si synchronizuje hodiny ze signálu). Nevýhoda: potřebuje dvojnásobnou šířku pásma oproti NRZ.

Příklad 0x01 0x05 v manchesterském kódování:
0x01 = 0000 0001, 0x05 = 0000 0101
Každý bit generuje přechod uprostřed periody. Bit 0: přechod ↓ (high→low), bit 1: přechod ↑ (low→high).
Výsledkem je signál s přechody u každého bitu — žádná ambiguita, žádná potřeba externího hodinového signálu.

Další kódovací schémata

  • NRZI (Non-Return to Zero Inverted): změna napětí = 1, beze změny = 0. USB, FDDI.
  • 4B/5B: každé 4 bity dat kódovány do 5 bitů — garantuje dostatek přechodů. Používá se v 100BASE-TX (Fast Ethernet).
  • 8B/10B: 8 bitů → 10 bitů; Gigabit Ethernet, Fibre Channel.
  • PAM4: 4 úrovně napětí → 2 bity na symbol; moderní 100G/400G Ethernet.
Intuice: Kódovací schéma řeší problém "jak odlišit 000...0 od klidového stavu linky?" Manchesterské kódování to řeší elegantně — přechod vždy existuje, takže receiver vždy ví, kdy přišel nový bit.
Shrnutí:
  • NRZ: jednoduché, ale ztráta synchronizace při dlouhých sériích stejných bitů.
  • Manchester: přechod uprostřed každého bitu → samosynchronizace; 2× šířka pásma.
  • 4B/5B, 8B/10B: kompromis — overhead kódování, ale zaručené přechody.
  • PAM4: víceúrovňové kódování pro vysoké rychlosti (100G+).

13. Kvalitativní parametry přenosu

Jak poznáme, zda síť funguje dobře? Potřebujeme měřitelné parametry, které popisují kvalitu přenosu na různých vrstvách modelu OSI.

Fyzická vrstva (vrstva 1)

  • BER (Bit Error Rate): poměr chybně přijatých bitů k celkovému počtu; typicky 10⁻⁹ nebo lepší pro Ethernet.
  • Šířka pásma: maximální kapacita přenosového kanálu [bps].
  • Útlum: zeslabení signálu [dB]; měří se na optice OTDR.
  • SNR (Signal-to-Noise Ratio): poměr signálu k šumu — čím vyšší, tím lepší.

Spojová vrstva (vrstva 2)

  • FER (Frame Error Rate): podíl poškozených nebo zahozených rámců.
  • Propustnost (throughput): skutečně přenesená data za jednotku času (netto, bez overhead rámcování).
  • Kolizní doména: u historického Ethernet — míra kolizí ovlivňuje efektivní propustnost.

Síťová vrstva (vrstva 3)

  • Latence (zpoždění): čas od odeslání paketu po přijetí odpovědi (round-trip time, RTT); měří se pomocí ICMP ping.
  • Jitter: variabilita zpoždění — kritická pro VoIP a video streaming.
  • Packet loss: procento ztracených paketů — ovlivňuje TCP propustnost.
  • Hop count: počet routerů na cestě — indikátor složitosti trasy.
Nástroje: ping měří RTT a packet loss. traceroute/tracert zobrazí cestu (hop by hop) a RTT na každém uzlu. OTDR (Optical Time Domain Reflectometer) měří útlum a zlomy v optickém vlákně.
Shrnutí:
  • Fyzická: BER, SNR, šířka pásma, útlum.
  • Spojová: FER, throughput, kolize.
  • Síťová: latence (RTT), jitter, packet loss, hop count.
  • ping = RTT + ztrátovost; traceroute = vizualizace trasy.

14. Ethernet – standard, struktura rámce, funkce switche

Ethernet (IEEE 802.3) je dnes dominantní standard pro drátové LAN. Vznikl v 70. letech v Xerox PARC a prošel vývojem od 10 Mbit/s coaxiálu až po dnešní 400 Gbit/s optiku.

Definice: Ethernet je rodina síťových technologií definovaná standardem IEEE 802.3, pracující na fyzické a linkové vrstvě OSI. Specifikuje formát rámce, přístupovou metodu (CSMA/CD) a fyzická média.

Evoluce Ethernetu

100BASE-TX100 Mbit/sUTP Cat5Fast Ethernet, kódování 4B/5B

Struktura Ethernet rámce (viz téma 6)

Maximální velikost rámce (MTU) = 1500 bajtů payload + 14 bajtů hlavička + 4 bajty FCS = 1518 bajtů. Jumbo frames mohou mít MTU až 9000 bajtů (datacentra).

Funkce switche v Ethernet síti

Moderní Ethernet je plně přepínaný (switched): každý port switche je vlastní kolizní doménou, všechny porty jsou v full-duplex režimu. Switch (vrstva 2) přeposílá rámce na základě MAC tabulky. L3 switch navíc umí routing mezi VLANy.

Shrnutí:
  • Ethernet = IEEE 802.3; dnes 1G–400G; od koaxiálu přes UTP po optiku.
  • Rámec: preambule | dst MAC | src MAC | EtherType | payload (46–1500 B) | FCS.
  • Moderní Ethernet = full duplex + switched → CSMA/CD prakticky nepoužívaný.
  • L3 switch = switch + routing = kombinace vrstvy 2 a 3.

16. TCP vs. UDP – porovnání vlastností

Transportní vrstva nabízí dva základní protokoly. Volba závisí na tom, co aplikace potřebuje: spolehlivost nebo rychlost.

VlastnostTCPUDP
SpojeníSpojově orientovaný (handshake)Bezspojový (connectionless)
SpolehlivostGarantováno pořadí, žádné ztrátyŽádná záruka (best-effort)
OverheadVyšší (hlavička 20+ B, ACK, handshake)Nižší (hlavička 8 B)
RychlostPomalejší (flow/congestion control)Rychlejší (žádné čekání na ACK)

Intuice: TCP je jako doporučená pošta — potvrzení doručení, ale pomalejší. UDP je jako pohlednice — rychlejší, ale nevíš, jestli dorazila. VoIP raději přijme výpadek jednoho paketu (drobný šum) než aby čekalo na retransmisi (sekundové okno ticha).
Proč DNS používá UDP: Dotaz i odpověď jsou krátké (vejdou se do jednoho datagramu), rychlost je důležitá, a pokud odpověď nepřijde, stačí zopakovat dotaz — bez potřeby TCP handshaku.
Pozor: UDP sám o sobě nezaručuje nic, ale aplikace mohou implementovat vlastní spolehlivost nad UDP (např. QUIC protokol, RTP s RTCP). V takovém případě mají výhody obou světů.
Shrnutí:
  • TCP = spolehlivý, seřazený, řízený; vyšší overhead; web, e-mail, SSH.
  • UDP = rychlý, bezspojový, no guarantees; DNS, VoIP, streaming, gaming.
  • Obojí identifikuje aplikace pomocí čísel portů (16 bitů, 0–65535).
  • QUIC = moderní protokol nad UDP implementující spolehlivost (HTTP/3).

17. Prvky síťové infrastruktury – router, switch, hub

V síti existují aktivní prvky, které přeposílají nebo zesilují data. Liší se tím, na jaké vrstvě OSI pracují a jak "inteligentně" rozhodují o přeposílání.

PrvekVrstva OSIAdresaceFunkce

Switch (přepínač)2 – LinkováMACPřeposílání rámců na základě MAC tabulky. Každý port = vlastní kolizní doména. Standard v LAN.

Klíčový princip: Vyšší vrstva = větší inteligence = větší latence, ale chytřejší rozhodování. Hub nic nerozmýšlí (naivní broadcast), switch se dívá na MAC (lokální inteligence), router zná celou topologii internetu.
Shrnutí:
  • Hub (L1): vše na všechny porty, zastaralý, jedna kolizní doména.
  • Switch (L2): přepíná podle MAC tabulky, každý port = kolizní doména.
  • Router (L3): směruje podle IP, odděluje broadcast domény, propojuje sítě.
  • L3 switch = switch + router v jednom zařízení (fast path hardware routing).

  • Defense in depth = vrstvená ochrana kombinující šifrování, filtrování a monitoring.

19. Přepínané sítě, VLAN – virtualizace

Fyzická síť je jedna — ale logicky ji chceme rozdělit. VLAN (Virtual LAN) umožňuje na jedné fyzické infrastruktuře provozovat více izolovaných sítí.

Definice: VLAN je logická skupina portů (nebo zařízení) na switchi, která tvoří vlastní broadcast doménu nezávisle na fyzickém umístění. Zařízení v různých VLAN nemohou komunikovat přímo — potřebují router nebo L3 switch.

Tagování IEEE 802.1Q

Standard 802.1Q definuje vložení VLAN tagu (4 bajty) do Ethernet rámce za zdrojovou MAC adresu. Tag obsahuje VID (VLAN ID) — 12 bitů → 4094 různých VLAN (0 a 4095 rezervovány).

Access port vs. Trunk port:
  • Access port: připojeno koncové zařízení (PC, tiskárna). Port patří jedné VLAN. Příchozí rámce se tagují interně; odchozí se odesílají bez tagu (device to nezná VLAN).
  • Trunk port: propojení switch-switch nebo switch-router. Přenáší rámce více VLAN zároveň s 802.1Q tagy. Router pak provádí inter-VLAN routing ("router on a stick").

Proč VLAN?

  • Bezpečnostní izolace: VLAN pro HR, VLAN pro Finance, VLAN pro hosty — navzájem nemohou komunikovat.
  • Snížení broadcast domény: broadcast flood se šíří jen uvnitř VLAN, ne do celé sítě.
  • Flexibilita: přesun zařízení do jiné VLAN rekonfigurací portu, bez fyzického přepojení.
Shrnutí:
  • VLAN = logická izolace na L2 — vlastní broadcast doména na sdílené fyzické infrastruktuře.
  • IEEE 802.1Q: 4bajtový tag s VID (12 bitů, 4094 VLAN) vložený do Ethernet rámce.
  • Access port = jedno VLAN, bez tagů ven; trunk port = více VLAN, tagy zachovány.
  • Inter-VLAN routing potřebuje router nebo L3 switch.

20. IPv4 vs. IPv6 – rozdíly, vzájemná komunikace

IPv4 adresní prostor (4,3 miliardy adres) je vyčerpán. IPv6 byl navržen jako nástupce s obrovskými adresním prostorem a dalšími vylepšeními.

Vzájemná komunikace (přechodové mechanismy)

  • Dual Stack: zařízení má obě adresy (IPv4 i IPv6) a komunikuje oběma protokoly. Nejjednodušší přechod.
  • Tunneling: IPv6 pakety zapouzdřeny do IPv4 paketů pro přenos přes IPv4 sítě (6in4, 6to4, Teredo).

Shrnutí:
  • IPv6 = 128 bitů, 3,4×10³⁸ adres, fixní 40B header, bez NAT a broadcastu.
  • Přechod: dual stack (obě adresy zároveň) je nejjednodušší.
  • NAT64: IPv6 sítě komunikují s IPv4 servery přes překladač.
  • SLAAC: IPv6 zařízení si konfiguruje adresu samo z Router Advertisement.

21. DNS – princip fungování

IP adresy si nikdo nepamatuje. DNS (Domain Name System) překládá lidsky čitelná doménová jména (www.google.com) na IP adresy. Je to distribuovaný hierarchický databázový systém.

Definice: DNS je hierarchický, distribuovaný systém překladu doménových jmen na IP adresy (a naopak — reverzní DNS). Pracuje na aplikační vrstvě, primárně přes UDP port 53 (krátké dotazy) nebo TCP port 53 (velké odpovědi, zone transfery).

Hierarchie DNS

  • Root servery (13 clusterů, A–M): znají adresy TLD serverů.
  • TLD servery (Top-Level Domain): .com, .cz, .org — znají adresy autoritativních serverů pro domény.
  • Autoritativní servery: spravují konkrétní doménu (např. google.com), odpovídají finálně.
  • Rekurzivní resolver: provozovaný ISP nebo Googlem (8.8.8.8) — přijme dotaz od klienta, projde hierarchii a vrátí výsledek.

Průběh DNS dotazu (rekurzivní)

Příklad: Klient se ptá na www.example.com:
  1. Klient → Rekurzivní resolver: "Co je IP www.example.com?"
  2. Resolver → Root server: "Kdo spravuje .com?"
  3. Root server → Resolver: "TLD server pro .com je X."
  4. Resolver → TLD server: "Kdo spravuje example.com?"
  5. TLD server → Resolver: "Autoritativní server je Y."
  6. Resolver → Autoritativní server: "Co je IP www.example.com?"
  7. Autoritativní server → Resolver: "93.184.216.34"
  8. Resolver → Klient: "93.184.216.34" (a uloží do cache na TTL sekund)
Caching: DNS záznamy mají TTL (Time To Live). Resolver cachuje odpovědi → většina dotazů se vyřeší lokálně bez procházení celé hierarchie. Tímto se systém škáluje na miliardy dotazů denně.
Shrnutí:
  • DNS = hierarchický překladač jmen → IP adresy; UDP/TCP port 53.
  • Hierarchie: Root → TLD → Autoritativní server.
  • Rekurzivní resolver za klienta projde celou hierarchií a výsledek cachuje.
  • TTL určuje jak dlouho je záznam platný v cache — ovlivňuje propagaci změn.

22. Aplikační protokoly – SMTP, ICMP, ARP

Stručný přehled tří důležitých protokolů, které se pohybují na různých vrstvách a slouží různým účelům.

ARP – Address Resolution Protocol (Vrstva 2)

Definice: ARP mapuje IPv4 adresu na MAC adresu v rámci lokální sítě. Každé zařízení udržuje ARP cache (IP → MAC).

Princip: Chci poslat paket na IP 192.168.1.5, ale neznám jeho MAC. Vyšlu broadcast ARP Request: "Kdo má IP 192.168.1.5?" Zařízení s touto IP odpoví unicastem: "Já! Moje MAC je XX:XX." Zapíši do ARP cache.

ICMP – Internet Control Message Protocol (Vrstva 3)

Definice: ICMP přenáší řídicí a chybové zprávy na síťové vrstvě. Je součástí IP (protokolové číslo 1), ale není transportní protokol — neposílá aplikační data.

Použití: ping = ICMP Echo Request + Echo Reply (měření RTT a dostupnosti). traceroute využívá ICMP Time Exceeded (TTL expiraci). Routery oznamují chyby pomocí ICMP Destination Unreachable.

SMTP – Simple Mail Transfer Protocol (Vrstva 7)

Definice: SMTP je aplikační protokol pro odesílání e-mailu mezi mail servery (TCP port 25, nebo 587 pro submission). Pro příjem e-mailů klientem se používá IMAP (port 993) nebo POP3.

Princip: Klient se připojí k SMTP serveru, projde handshake (EHLO), autentizuje se, specifikuje odesílatele (MAIL FROM), příjemce (RCPT TO) a tělo zprávy (DATA). Server zprávu přijme a doručí cílovému SMTP serveru příjemce (zjistí ho z DNS MX záznamu).

Shrnutí:
  • ARP (L2): IP → MAC překlad; broadcast dotaz, unicast odpověď, cache s TTL.
  • ICMP (L3): řídicí protokol IP; ping (Echo), traceroute (TTL exceeded), chybové hlášení.
  • SMTP (L7): odesílání e-mailů; port 25/587; příjem = IMAP/POP3.

23. IPSec – základní funkční principy

IPSec (Internet Protocol Security) je rozšíření IP protokolu zajišťující bezpečnost na síťové vrstvě. Pracuje transparentně — aplikace nemusí být upravovány, bezpečnost je "pod" nimi.

Definice: IPSec je sada protokolů (RFC 4301) pro šifrování, autentizaci a integritu IP komunikace na vrstvě 3. Může chránit jak individuální pakety, tak celé VPN tunely.

Tři klíčové protokoly

  • AH (Authentication Header): zajišťuje integritu a autentizaci odesílatele, ale nešifruje payload. Chrání celý paket včetně IP hlavičky. Prakticky obsoletní — neprojde NATem (mění IP hlavičku → AH hash selže).
  • ESP (Encapsulating Security Payload): šifruje payload + zajišťuje integritu a autentizaci. Nevztahuje se na vnější IP hlavičku. Standardní volba pro VPN.
  • IKE (Internet Key Exchange): protokol pro automatické vyjednání klíčů a parametrů SA (Security Association). IKEv2 je modernější, odolnější vůči útokům.

Provozní módy

MódCo chráníPoužití
Transportní módPouze payload (data) — IP hlavička zůstává viditelnáEnd-to-end ochrana mezi dvěma hostiteli
Tunelový módCelý původní paket (IP + data) zapouzdřen do nového IP paketuVPN gateway-to-gateway (site-to-site VPN)
Site-to-site VPN s IPSec: Router pobočky A (GW-A) a router pobočky B (GW-B) vyjednají přes IKE SA (šifrovací algoritmus, klíče). Pakety z A do B se zabalí do ESP tunelového módu, odešlou po internetu šifrovaně, GW-B rozbalí a doručí do vnitřní sítě B.
Shrnutí:
  • IPSec = bezpečnost na L3; transparentní pro aplikace.
  • AH = autentizace (bez šifrování); ESP = šifrování + autentizace. V praxi ESP.
  • IKE = automatická výměna klíčů a vyjednání SA.
  • Tunelový mód = celý paket zapouzdřen → VPN; transportní mód = jen payload → end-to-end.

24. TLS – základní funkční principy

TLS (Transport Layer Security) je nástupce SSL a zajišťuje bezpečnou komunikaci na aplikační vrstvě. Je základem HTTPS, SMTP over TLS, IMAPS a dalších bezpečných protokolů.

Definice: TLS je kryptografický protokol (RFC 8446 pro TLS 1.3) zajišťující důvěrnost (šifrování), integritu (MAC) a autentizaci (certifikáty) komunikace nad TCP. Pracuje mezi transportní a aplikační vrstvou.

TLS Handshake (TLS 1.3)

  1. ClientHello: Klient pošle podporované verze TLS, šifrovací sady (cipher suites), a klíčové sdílení (key share) pro Diffie-Hellman.
  2. ServerHello + Certificate: Server zvolí cipher suite, pošle svůj X.509 certifikát a vlastní DH klíčový podíl.
  3. Ověření certifikátu: Klient ověří certifikát serveru pomocí PKI (chain of trust → kořenová CA). Ověří, zda certifikát odpovídá doméně a není revokován.
  4. Výměna klíčů (DH): Obě strany odvodí sdílený tajný klíč (pre-master secret → master secret) bez jeho přenosu sítí. To zajišťuje perfect forward secrecy.
  5. Finished: Obě strany potvrdí handshake. Od této chvíle komunikace probíhá symetrickým šifrováním (AES-GCM).
Proč asymetrické → symetrické? Asymetrické šifrování (RSA, DH) je bezpečné pro výměnu klíčů, ale pomalé pro velké objemy dat. TLS ho používá jen v handshaku pro vyjednání symetrického klíče. Symetrickým šifrováním (AES) se pak šifruje veškerý datový tok — je 100× rychlejší.
Perfect Forward Secrecy (PFS): Každá TLS session používá nové, dočasné DH klíče. I kdyby byl kompromitován privátní klíč serveru, nelze dešifrovat historicky zachycené komunikace. TLS 1.3 mandatorně vyžaduje PFS (odstranil statické RSA výměny klíčů).
Pozor: TLS 1.0 a 1.1 jsou zastaralé a zranitelné (POODLE, BEAST, BEAST). TLS 1.2 je akceptovatelné. TLS 1.3 (2018) je doporučený standard — zjednodušil handshake a odstranil slabé algoritmy.
Shrnutí:
  • TLS zajišťuje důvěrnost (šifrování), integritu (MAC) a autentizaci (certifikáty X.509).
  • Handshake: DH výměna klíčů → sdílený symetrický klíč bez přenosu po síti.
  • Perfect Forward Secrecy: dočasné klíče každé session → historický provoz nelze dešifrovat.
  • TLS 1.3 = aktuální standard; TLS 1.0/1.1 zastaralé a zranitelné.
