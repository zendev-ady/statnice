Databázové systémy

1. Transakce – definice, ACID vlastnosti

Transakce je logická jednotka práce s databází – buď se provede celá, nebo vůbec. Je to jako bankovní převod: buď peníze odejdou z účtu A a přijdou na účet B, nebo se nestane nic.

Definice: Transakce je posloupnost databázových operací (čtení, zápis), která přechází databázi z jednoho konzistentního stavu do druhého. Transakce začíná příkazem BEGIN a končí buď COMMIT (potvrzení) nebo ROLLBACK (zrušení).

ACID vlastnosti

Každá správně implementovaná transakce musí splňovat čtyři vlastnosti:

A – Atomicita (Atomicity): Transakce je nedělitelná. Buď se provedou všechny operace, nebo žádná. Pokud dojde k chybě uprostřed transakce, databáze se vrátí do původního stavu (rollback).
C – Konzistence (Consistency): Transakce přechází databázi z jednoho konzistentního stavu do druhého. Nikdy nesmí porušit integritní omezení (cizí klíče, CHECK podmínky apod.).
I – Izolace (Isolation): Souběžně běžící transakce se navzájem neovlivňují — každá transakce "vidí" databázi, jako by běžela sama. V praxi se stupeň izolace nastavuje (viz téma 8).
D – Trvanlivost (Durability): Jednou potvrzená (COMMIT) transakce je trvale uložena — i při výpadku systému. Zajišťuje to WAL (Write-Ahead Log / transakční žurnál).
Příklad: Převod 1000 Kč z účtu A na účet B:
  1. UPDATE ucty SET zustatek = zustatek - 1000 WHERE id = 'A'
  2. UPDATE ucty SET zustatek = zustatek + 1000 WHERE id = 'B'
  3. COMMIT
Pokud selže krok 2, atomicita zajistí, že se odroluje i krok 1. Peníze nezmizí.
Upozornění: Konzistence v ACID není totéž co konzistence v CAP teorému (viz NoSQL). V ACID jde o konzistenci vůči integritním omezením databáze.
Shrnutí:
  • Transakce = nedělitelná jednotka práce, končí COMMIT nebo ROLLBACK.
  • Atomicita: vše nebo nic. Conzistence: zachování pravidel DB. Izolace: transakce se nevidí. Durability: potvrzená data přežijí pád.
  • Implementuje se přes WAL (žurnál) a zamykací mechanismy.

2. SQL – základní příkazy, struktura dotazu

SQL (Structured Query Language) je deklarativní jazyk pro práci s relačními databázemi — říkáme co chceme, ne jak to získat.

Kategorie příkazů

KategoriePříkazyÚčel
DMLSELECT, INSERT, UPDATE, DELETEManipulace s daty
DDLCREATE, ALTER, DROP, TRUNCATEDefinice struktury
DCLGRANT, REVOKEŘízení přístupu
TCLBEGIN, COMMIT, ROLLBACKSpráva transakcí

SELECT – logické pořadí zpracování

Pořadí zpracování (logické, ne pořadí psaní):
  1. FROM + JOIN — určení zdrojových tabulek
  2. WHERE — filtrování řádků
  3. GROUP BY — seskupení
  4. HAVING — filtrování skupin
  5. SELECT — výběr sloupců
  6. ORDER BY — řazení výsledku
  7. LIMIT / OFFSET — stránkování
Příklad INSERT / UPDATE / DELETE: 
-- Vložení nového záznamu
INSERT INTO studenti (jmeno, rocnik) VALUES ('Jana', 2);

-- Aktualizace
UPDATE studenti SET rocnik = 3 WHERE jmeno = 'Jana';

-- Smazání
DELETE FROM studenti WHERE rocnik = 1;
Upozornění: DELETE FROM tabulka bez WHERE smaže všechny řádky! TRUNCATE dělá totéž rychleji, ale nelze ho rolovat zpět v některých DB.
Shrnutí:
  • SQL = deklarativní jazyk; říkáme CO, ne JAK.
  • DML pracuje s daty, DDL definuje strukturu, TCL řídí transakce.
  • Logické pořadí zpracování SELECT: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY.

3. Normální formy (1NF, 2NF, 3NF, BCNF)

Normalizace je proces návrhu databáze tak, aby se minimalizovala redundance dat a předcházelo anomáliím při vkládání, aktualizaci a mazání.

1NF (První normální forma): Každý atribut obsahuje pouze atomické (nedělitelné) hodnoty a každý řádek je jednoznačně identifikovatelný. Žádné opakující se skupiny ani vícenásobné hodnoty v jednom sloupci (seznam telefonů v jednom poli).
2NF (Druhá normální forma): Je v 1NF a každý neklíčový atribut je plně funkčně závislý na celém primárním klíči. Problém nastává jen u složených klíčů — pokud atribut závisí jen na části klíče, tabulka není v 2NF.
3NF (Třetí normální forma): Je v 2NF a neexistuje tranzitivní závislost neklíčového atributu na klíči (tj. A→B a B→C, kde B není klíč). Všechny neklíčové atributy závisí přímo na klíči, celém klíči a jen na klíči.
BCNF (Boyce-Coddova normální forma): Přísnější verze 3NF. Pro každou netriviální funkční závislost $X \to Y$ musí být $X$ nadklíčem. BCNF řeší anomálie, které 3NF nestačí odstranit (nastává u relací s více překrývajícími se kandidátními klíči).
Příklad porušení 2NF:
Tabulka: (student_id, předmět_id, jméno_studenta, název_předmětu, hodnocení)
PK: (student_id, předmět_id)
Problém: jméno_studenta závisí jen na student_id, ne na celém klíči.
Oprava: Rozdělit na Studenti(student_id, jmeno), Předměty(predmet_id, nazev) a Zápisy(student_id, predmet_id, hodnoceni).
Příklad porušení 3NF:
Tabulka: (zamestnanec_id, oddeleni_id, nazev_oddeleni)
Problém: zamestnanec_id → oddeleni_id → nazev_oddeleni — tranzitivní závislost.
Oprava: Oddělit Oddělení(oddeleni_id, nazev).
Upozornění: Normalizace není vždy žádoucí — v analytických databázích (OLAP, datové sklady) se záměrně denormalizuje pro výkon čtení.
Shrnutí:
  • 1NF: atomické hodnoty, žádné opakující se skupiny.
  • 2NF: žádná částečná závislost na části složeného klíče.
  • 3NF: žádná tranzitivní závislost neklíčového atributu.
  • BCNF: každý determinant (levá strana FZ) musí být nadklíč.

4. Typy JOINů a jejich optimalizace

JOIN spojuje řádky ze dvou nebo více tabulek na základě společné podmínky. Klíčový nástroj pro práci s normalizovanými daty.

INNER JOIN: Vrátí pouze řádky, kde existuje shoda v obou tabulkách. Nejčastější typ.
LEFT (OUTER) JOIN: Vrátí všechny řádky z levé tabulky + odpovídající z pravé. Kde shoda neexistuje, doplní NULL.
RIGHT (OUTER) JOIN: Symetricky — všechny řádky z pravé tabulky + NULL pro nespárované z levé.
FULL (OUTER) JOIN: Vrátí všechny řádky z obou tabulek. Nespárované doplní NULL.
CROSS JOIN: Kartézský součin — každý řádek levé tabulky s každým řádkem pravé. Výsledek má $m \times n$ řádků.
Příklad: 
-- Studenti a jejich zápisy (jen kdo má zápis)
SELECT s.jmeno, z.predmet
FROM Studenti s INNER JOIN Zapisy z ON s.id = z.student_id;

-- Všichni studenti, i bez zápisu
SELECT s.jmeno, z.predmet
FROM Studenti s LEFT JOIN Zapisy z ON s.id = z.student_id;

Optimalizace a paměťová náročnost

AlgoritmusSložitostKdy se použije
Nested Loop Join$O(n \cdot m)$Malá tabulka + index na druhé
Hash Join$O(n + m)$Velké tabulky bez vhodného indexu
Merge Join$O(n \log n + m \log m)$Obě tabulky seřazeny / indexovány

Paměťová náročnost Hash Joinu: Menší tabulka se načte do hash tabulky v RAM (build phase), větší se prochází (probe phase). Pokud se hash tabulka nevejde do paměti, dochází k hash spill na disk — výrazné zpomalení.

Shrnutí:
  • INNER = průnik; LEFT/RIGHT = vše z jedné strany + NULL; FULL = vše z obou.
  • Optimalizátor volí Nested Loop (s indexem), Hash Join (velká data) nebo Merge Join (seřazená data).
  • Hash Join je paměťově náročný — celá menší tabulka musí vejít do RAM.

5. SELECT – detailní popis, agregační funkce

SELECT je nejpoužívanější SQL příkaz — slouží k výběru, filtrování, seskupení a agregaci dat.

Základní struktura SELECT: 
SELECT [DISTINCT] sloupce / výrazy / agregace
FROM   tabulka
[JOIN  ...]
[WHERE podmínka]          -- filtruje řádky před seskupením
[GROUP BY sloupce]        -- seskupí řádky do skupin
[HAVING podmínka]         -- filtruje skupiny (po GROUP BY)
[ORDER BY sloupce [ASC|DESC]]
[LIMIT n OFFSET m];

Agregační funkce

FunkcePopisChování pro NULL
COUNT(*)Počet řádků včetně NULLPočítá všechny řádky
COUNT(sloupec)Počet nenullových hodnotIgnoruje NULL
SUM(sloupec)Součet hodnotIgnoruje NULL
AVG(sloupec)Průměr hodnotIgnoruje NULL (nedělí jimi)
MIN / MAXMinimum / MaximumIgnoruje NULL
Příklad – průměrné hodnocení na předmět, jen lepší než 2.0: 
SELECT predmet, COUNT(*) AS pocet, AVG(hodnoceni) AS prumer
FROM Zapisy
WHERE hodnoceni IS NOT NULL
GROUP BY predmet
HAVING AVG(hodnoceni) > 2.0
ORDER BY prumer ASC;
WHERE vs HAVING: WHERE filtruje před seskupením (pracuje s jednotlivými řádky), HAVING filtruje po seskupení (pracuje s agregáty). Agregační funkce nelze použít ve WHERE.
Upozornění: Sloupce v SELECT, které nejsou v GROUP BY, musí být obaleny agregační funkcí — jinak dotaz selže (striktní DB) nebo vrátí nedefinovaný výsledek.
Shrnutí:
  • Pořadí zpracování: FROM → WHERE → GROUP BY → HAVING → SELECT → ORDER BY.
  • Agregační funkce (COUNT, SUM, AVG, MIN, MAX) ignorují NULL.
  • HAVING = WHERE pro skupiny; WHERE = filtr pro jednotlivé řádky.

6. Relační model – základy, ER diagramy (konceptuální model)

Relační model je způsob organizace dat do tabulek (relací) propojených klíči. ER diagram je vizuální nástroj pro konceptuální návrh — modeluje doménu před implementací v konkrétní DB.

Relace (tabulka): Množina n-tic (řádků) se stejnou strukturou. Každý sloupec je atribut, každý řádek je n-tice. Pořadí řádků i sloupců je matematicky nedefinované (jde o množinu).

ER Diagram – komponenty

SymbolPopis
ObdélníkEntita (věc/objekt, např. Student, Předmět)
ElipsaAtribut entity (jméno, rodné číslo...)
Podtržená elipsaIdentifikující atribut (primární klíč)
KosočtverecVztah mezi entitami (Zapisuje se, Pracuje v...)
Čára s čísly / písmenyKardinalita vztahu (1:1, 1:N, M:N)
Příklad: Student zapisuje se na Předmět (M:N vztah).
ER: entita Student — kosočtverec „Zapise_se" — entita Předmět, kardinalita M:N.
Převod do relačního modelu: vznik vazební tabulky Zapis(student_id FK, predmet_id FK, hodnoceni).
Upozornění: Slovo „relace" v relačním modelu neznamená vztah — označuje tabulku (z lat. relatio = uspořádaná množina). Vztah v ER diagramu se anglicky jmenuje relationship.
Shrnutí:
  • Relace = tabulka; atribut = sloupec; n-tice = řádek.
  • ER diagram modeluje entity, jejich atributy a vztahy (kardinalitu) — nezávisle na DB.
  • M:N → vazební tabulka; 1:N → cizí klíč na straně N.

7. Dirty read a problémy paralelního přístupu (RW, WR, WW konflikty)

Když více transakcí běží souběžně, mohou si vzájemně "šlapat na prsty" — vznikají anomálie v čtení a zápisu dat.

Tři čtecí anomálie

Dirty Read (špinavé čtení): Transakce T2 přečte data, která T1 změnila, ale ještě nepotvrdila (COMMIT). Pokud T1 provede ROLLBACK, T2 pracuje s daty, která nikdy neexistovala.
Non-Repeatable Read (neopakovatelné čtení): T2 přečte stejný řádek dvakrát, ale dostane jiné výsledky, protože T1 mezitím řádek změnila a potvrdila.
Phantom Read (fantómové čtení): T2 provede stejný dotaz dvakrát, ale dostane jiný počet řádků, protože T1 mezitím řádky vložila nebo smazala a potvrdila.

Konflikty operací (WR, RW, WW)

KonfliktPopisAnomálie
WRT1 zapíše (nepotvrzeno), T2 čteDirty Read
RWT1 čte, T2 zapíše a potvrdí, T1 čte znovuNon-Repeatable Read
WWT1 zapíše, T2 přepíše, T1 odroluje → ztráta T2Lost Update
Příklad dirty read:
T1: UPDATE ucet SET zustatek = 0 WHERE id=1 (nepotvrzeno)
T2: SELECT zustatek FROM ucet WHERE id=1 → vrátí 0
T1: ROLLBACK — zůstatek je zpět na původní hodnotě
T2 pracovala s hodnotou, která reálně nikdy neexistovala.
Shrnutí:
  • Dirty read: čtu nepotvrzená data (WR konflikt).
  • Non-repeatable read: stejný řádek dá jiný výsledek (RW konflikt).
  • Phantom read: stejný dotaz dá jiný počet řádků (nové vložení/smazání).
  • WW konflikt → ztráta aktualizace (Lost Update).

8. Zamykání, stupně izolovanosti

Zamykání je hlavní mechanismus, jak databáze zajišťuje izolaci transakcí. Stupně izolace jsou kompromis mezi bezpečností a výkonem.

Typy zámků

Sdílený zámek (S-lock / Read lock): Více transakcí může současně číst. Nikdo nesmí psát, dokud jsou S-locky aktivní.
Exkluzivní zámek (X-lock / Write lock): Pouze jedna transakce může psát. Žádná jiná transakce nesmí ani číst, ani psát.

Stupně izolovanosti (SQL standard)

StupeňDirty ReadNon-Rep. ReadPhantom Read
READ UNCOMMITTEDmožnýmožnýmožný
READ COMMITTED✗ zamezenomožnýmožný
REPEATABLE READ✗✗ zamezenomožný
SERIALIZABLE✗✗✗ zamezeno
Příklad READ COMMITTED (výchozí v PostgreSQL):
Transakce vidí jen potvrzená data jiných transakcí. Řeší dirty read, ale ne non-repeatable read — dvě čtení ve stejné transakci mohou vrátit různé výsledky, pokud jiná transakce mezitím změnila data.
Upozornění: Vyšší stupeň izolace = méně anomálií, ale nižší výkon (více uzamčení, více čekání, více deadlocků). READ UNCOMMITTED se v praxi téměř nepoužívá.
Shrnutí:
  • S-lock: sdílené čtení; X-lock: exkluzivní zápis.
  • READ COMMITTED (výchozí) zabrání dirty read.
  • SERIALIZABLE zabrání všem anomáliím za cenu výkonu.

9. Funkční závislosti – definice, klíč, nadklíč

Funkční závislosti popisují, jaké atributy "určují" jiné atributy — jsou matematickým základem normalizace databáze.

Funkční závislost (FZ): Říkáme, že $X \to Y$ (X funkčně určuje Y), pokud pro každé dvě n-tice $t_1, t_2$ v relaci platí: pokud $t_1[X] = t_2[X]$, pak $t_1[Y] = t_2[Y]$. Znalost hodnot atributů X jednoznačně určuje hodnotu atributů Y.
Nadklíč (Superkey): Množina atributů $K$, která funkčně určuje všechny ostatní atributy relace: $K \to \text{(všechny atributy)}$. Může obsahovat "nadbytečné" atributy.
Kandidátní klíč (Candidate Key): Minimální nadklíč — žádná vlastní podmnožina $K$ není nadklíčem. Relace může mít více kandidátních klíčů.
Primární klíč: Jeden zvolený kandidátní klíč, který slouží jako "oficiální" identifikátor řádků v tabulce.
Příklad: Tabulka Zamestnanec(rodne_cislo, email, jmeno, oddeleni)
FZ: rodne_cislo → {email, jmeno, oddeleni}, email → {rodne_cislo, jmeno, oddeleni}
Kandidátní klíče: {rodne_cislo}, {email}
Nadklíč: {rodne_cislo, email} (nadbytečný — samotné rodné číslo stačí)
Shrnutí:
  • $X \to Y$: znalost X jednoznačně určuje Y.
  • Nadklíč: určuje vše; kandidátní klíč: minimální nadklíč; primární klíč: zvolený klíč.
  • FZ jsou základem normalizace — pomáhají odhalit redundanci a závislosti.

10. ORM (Object-Relational Mapping) – principy, mapování vztahů

ORM překlenuje propast mezi objektovým světem programu a relačním světem databáze — píšeme kód v objektech, ORM se stará o SQL automaticky.

ORM (Object-Relational Mapping): Vrstva, která automaticky mapuje objekty (třídy) v kódu na tabulky v databázi a naopak. Vývojář pracuje s objekty, ORM generuje SQL dotazy.

Principy mapování

OO světRelační svět
Třída (class Student)Tabulka (studenti)
Atribut (jmeno)Sloupec (jmeno VARCHAR)
Objekt (instance)Řádek v tabulce
Reference (@ManyToOne)Cizí klíč
Kolekce (@OneToMany)Vazební tabulka / FK
Výhody ORM: Méně SQL kódu, přenositelnost mezi DB, práce s objekty je přirozenější.
Problémy ORM:
  • N+1 problém: Načtení kolekce provede 1 dotaz na seznam + N dotazů na každý prvek. Řeší se eager loading (JOIN FETCH).
  • Impedance mismatch: Objekty mají dědičnost, polymorfismus — tabulky ne. Mapování dědičnosti je složité.
  • Výkon: Generované SQL nemusí být optimální; pro složité dotazy je lepší vlastní SQL (JPQL, native query).
Shrnutí:
  • ORM = automatický překlad třída↔tabulka, objekt↔řádek.
  • Zjednodušuje vývoj, ale může způsobit N+1 problém a nedostatečný výkon.
  • Populární ORM: Hibernate (Java/JPA), SQLAlchemy (Python), Entity Framework (.NET).

11. Serializovatelnost – konfliktní páry, konfliktní uspořádání

Serializovatelnost garantuje, že výsledek souběžného provádění transakcí je stejný, jako by se prováděly po jedné — je to formální základ korektnosti rozvrhu.

Rozvrh (Schedule): Prokládané pořadí operací více transakcí. Sériový rozvrh: transakce se neprolínají (T1 celá, pak T2 celá). Serializovatelný rozvrh: jeho výsledek je ekvivalentní nějakému sériovému rozvrhu.
Konfliktní operace: Dvě operace jsou v konfliktu, pokud:
  1. Patří různým transakcím,
  2. Přistupují ke stejnému datovému objektu,
  3. Alespoň jedna z nich je zápis (W).
Typy: WR (write-read), RW (read-write), WW (write-write).
Konfliktní serializovatelnost a precedenční graf: Rozvrh je konfliktně serializovatelný, pokud ho lze přeuspořádat přehazováním nekonflikních operací na sériový rozvrh. Test: sestav precedenční graf — uzel = transakce, hrana $T_i \to T_j$ = Ti má konfliktní operaci před Tj.
Rozvrh je serializovatelný $\Leftrightarrow$ precedenční graf je acyklický (DAG).
Příklad: Rozvrh S: R1(A), R2(A), W1(A), W2(A)
Konflikt: W1(A) vs. R2(A) — R2 proběhl před W1 → hrana T2 → T1? Ne: W1 je po R2 → T1 zapsal po čtení T2 → hrana T1 → T2 pro W1 vs W2 (W1 před W2).
Graf: T1 → T2 (acyklický) → rozvrh je serializovatelný (ekvivalentní T1, T2).
Shrnutí:
  • Konflikt = různé transakce, stejný objekt, alespoň jeden zápis.
  • Precedenční graf: acyklický ↔ konfliktně serializovatelný.
  • 2PL protokol garantuje serializovatelné rozvrhy.

12. Armstrongovy axiomy, algoritmus nalezení všech klíčů

Armstrongovy axiomy jsou pravidla pro odvozování funkčních závislostí — umožňují z dané sady FZ odvodit všechny ostatní platné závislosti a najít klíče relace.

Tři základní axiomy (Armstrong, 1974):
  1. Reflexivita: Pokud $Y \subseteq X$, pak $X \to Y$. (Každá množina atributů určuje svou podmnožinu.)
  2. Augmentace (rozšíření): Pokud $X \to Y$, pak $XZ \to YZ$ pro libovolné $Z$. (Přidáme-li atributy na obě strany, závislost platí dál.)
  3. Tranzitivita: Pokud $X \to Y$ a $Y \to Z$, pak $X \to Z$.
Odvozená pravidla: sjednocení ($X \to Y, X \to Z \Rightarrow X \to YZ$) a dekompozice.

Uzávěr množiny atributů $X^+$

Algoritmus výpočtu uzávěru $X^+$:
  1. Začni s $X^+ = X$.
  2. Pro každou FZ $A \to B$ v množině F: pokud $A \subseteq X^+$, přidej $B$ do $X^+$.
  3. Opakuj krok 2, dokud se $X^+$ zvětšuje.
Pokud $X^+ = \text{všechny atributy}$, pak $X$ je nadklíč.

Algoritmus nalezení kandidátního klíče

  1. Začni s $K = \text{všechny atributy}$.
  2. Pro každý atribut $A \in K$: pokud $(K \setminus \{A\})^+ = \text{všechny atributy}$, odstraň $A$ z $K$.
  3. Výsledné $K$ je kandidátní klíč (minimální nadklíč).
Pro nalezení všech klíčů: vyzkoušej všechny podmnožiny a ověřuj uzávěrem.
Příklad: FZ: $\{AB \to C,\; C \to D,\; D \to A\}$, atributy $\{A,B,C,D\}$.
Uzávěr $\{AB\}^+$: AB→C, C→D, D→A → $\{A,B,C,D\}$ → nadklíč.
Odebrání A: $\{B\}^+ = \{B\}$ ≠ vše → A nelze odebrat.
Odebrání B: $\{A\}^+ = \{A,D\}$ ≠ vše → B nelze odebrat.
Kandidátní klíč: $\{A, B\}$.
Shrnutí:
  • 3 axiomy: reflexivita, augmentace, tranzitivita — úplné a bezesporné.
  • Uzávěr $X^+$: atributy, které X funkčně určuje (výpočet iterací).
  • $X$ je kandidátní klíč ↔ $X^+ = $ vše a žádná vlastní podmnožina toto nesplňuje.

13. Postup návrhu DB: konceptuální → relační model → ER

Návrh databáze probíhá ve třech fázích — od abstraktního popisu světa k fyzickému schématu tabulek.

1. Konceptuální model (ER diagram): Nezávislý na konkrétní DB. Modelujeme entity, jejich atributy a vztahy. Výstup: ER diagram s kardinalitami vztahů.
2. Logický (relační) model: Převod ER do tabulek. Pravidla:
  • Každá entita → tabulka s primárním klíčem.
  • Vztah 1:N → cizí klíč na straně N.
  • Vztah M:N → vazební tabulka s FK na obě strany.
  • Vztah 1:1 → FK v jedné z tabulek + UNIQUE, nebo sloučení.
V této fázi probíhá normalizace (1NF, 2NF, 3NF, BCNF).
3. Fyzický model: Volba datových typů, indexů, particionování. Závisí na konkrétním DBMS (PostgreSQL, MySQL, Oracle...).
Příklad celého procesu:
Konceptuální: Entity Student a Předmět s M:N vztahem „Zapisuje se".
Logický: Student(id PK, jmeno, email), Predmet(id PK, nazev, kredity), Zapis(student_id FK, predmet_id FK, rok, hodnoceni), PK(student_id, predmet_id).
Fyzický: Index na email, VARCHAR(100) pro jméno, NUMERIC(3,1) pro hodnocení.
Shrnutí:
  • Konceptuální (ER, nezávislý na DB) → Logický (relační schéma + normalizace) → Fyzický (implementace v konkrétní DB).
  • M:N vždy → vazební tabulka; 1:N → FK na straně N.
  • Normalizace se aplikuje v logické fázi před fyzickou implementací.

14. Integritní omezení – typy v SQL, referenční integrita, referenční akce

Integritní omezení jsou pravidla vymáhající korektnost a konzistenci dat přímo v databázi — chrání data bez ohledu na to, jaká aplikace s nimi pracuje.

OmezeníPopis
NOT NULLSloupec nesmí obsahovat NULL.
UNIQUEHodnoty musí být unikátní (NULL je povoleno vícekrát).
PRIMARY KEYNOT NULL + UNIQUE; jedinečná identifikace řádku.
FOREIGN KEYHodnota musí odpovídat existující hodnotě PK v jiné tabulce (nebo NULL).
CHECKPodmínka platná pro každý řádek (např. vek > 0).
DEFAULTVýchozí hodnota při nevyplnění sloupce.
Referenční integrita: Cizí klíč (FK) musí vždy odkazovat na existující primární klíč v referencované tabulce — nebo být NULL. Zabraňuje "osiřelým" záznamům.
Referenční akce (co se stane při smazání/změně referencovaného PK):
  • CASCADE: automaticky smaž/změň i závislé záznamy.
  • SET NULL: nastav FK na NULL.
  • SET DEFAULT: nastav FK na výchozí hodnotu.
  • RESTRICT / NO ACTION: zakáže operaci, pokud existují závislé záznamy.
Příklad: 
CREATE TABLE Zapisy (
  student_id INT REFERENCES Studenti(id) ON DELETE CASCADE,
  predmet_id INT REFERENCES Predmety(id) ON DELETE RESTRICT,
  hodnoceni  NUMERIC CHECK (hodnoceni BETWEEN 1 AND 4)
);
Smazání studenta → automaticky smaže jeho zápisy. Pokus o smazání předmětu, který má zápisy → zakázáno.
Shrnutí:
  • NOT NULL, UNIQUE, PK, FK, CHECK, DEFAULT — 6 základních omezení SQL.
  • FK + referenční akce (CASCADE, SET NULL, RESTRICT) zajišťují referenční integritu.
  • Omezení se kontrolují při každé DML operaci automaticky.

15. JPA – dědičnost objektů v DB, mapování vztahů, persistentní kontext, stavy entity

JPA (Jakarta Persistence API) je Java standard pro ORM. Definuje, jak mapovat objekty na tabulky, spravovat jejich životní cyklus a řešit dědičnost.

Stavy entity v JPA

4 stavy entity (životní cyklus):
  1. New (Transient): Objekt existuje v paměti, není spojen s DB ani EntityManagerem. Volání new Student().
  2. Managed (Persistent): EntityManager ho sleduje; změny se automaticky synchronizují s DB při commitu (dirty checking).
  3. Detached: Byl spravován, ale EntityManager byl uzavřen nebo se zavolalo detach(). Změny se nesynchronizují.
  4. Removed: Označen ke smazání; při commitu bude smazán z DB.
Persistentní kontext: Cache 1. úrovně, kterou spravuje EntityManager. Obsahuje všechny spravované entity v rámci jedné transakce/session. Při commitu synchronizuje změny s DB (flush). Každý EntityManager má svůj vlastní kontext.

Strategie mapování dědičnosti

StrategiePopisVýhody / Nevýhody
SINGLE_TABLECelá hierarchie v jedné tabulce + diskriminátorový sloupecRychlé čtení, žádný JOIN; NULL hodnoty pro nepoužité atributy
TABLE_PER_CLASSKaždá konkrétní třída má svou tabulkuŽádný JOIN pro konkrétní třídu; duplicita sloupců, obtížné polymorfní dotazy
JOINEDSdílené atributy v rodičovské tabulce, specifické ve vlastníNormalizovaný návrh; JOIN při každém čtení
Příklad SINGLE_TABLE: 
@Entity
@Inheritance(strategy = InheritanceType.SINGLE_TABLE)
@DiscriminatorColumn(name = "typ")
public class Vozidlo { int id; String znacka; }

@Entity @DiscriminatorValue("AUTO")
public class Auto extends Vozidlo { int pocetDveri; }

@Entity @DiscriminatorValue("MOTO")
public class Motorka extends Vozidlo { boolean sajdkara; }
Výsledek: jedna tabulka Vozidlo se sloupci id, znacka, typ, pocetDveri, sajdkara.
Shrnutí:
  • Stavy entity: New → Managed (em.persist) → Detached/Removed.
  • Persistentní kontext = cache EntityManageru; dirty checking auto-synchronizuje změny.
  • Dědičnost: SINGLE_TABLE (výkon), JOINED (normalizace), TABLE_PER_CLASS (izolace tabulek).

16. Vazby 1:1, 1:N, M:N – reprezentace v relačním a OO návrhu

Kardinalita vztahu určuje, kolik instancí jedné entity může být spojeno s instancemi druhé — a přímo ovlivňuje, jak vztah implementujeme v databázi.

1:1 (One-to-One): Každé instanci A odpovídá nejvýše jedna instance B a naopak.
Příklad: Osoba — Cestovní_pas (každý pas patří jedné osobě).
Implementace: FK v jedné z tabulek + UNIQUE constraint, nebo sloučení do jedné tabulky.
1:N (One-to-Many): Jedna instance A odpovídá více instancím B.
Příklad: Oddělení — Zaměstnanci (jedno oddělení má více zaměstnanců).
Implementace: FK na straně N — Zamestnanec.oddeleni_id → Oddeleni.id.
M:N (Many-to-Many): Více instancí A odpovídá více instancím B.
Příklad: Student — Předmět (student zapisuje více předmětů, předmět mají více studentů).
Implementace: Vazební (asociativní) tabulka s FK na obě strany. Může nést vlastní atributy (hodnocení, rok).
KardinalitaRelační modelJava/JPA
1:1FK + UNIQUE@OneToOne
1:NFK na straně N@OneToMany / @ManyToOne
M:NVazební tabulka@ManyToMany nebo entita pro vazbu
Upozornění: M:N vztah s vlastními atributy (např. hodnocení v Zápisu) by měl být modelován jako plnohodnotná entita, ne jen @ManyToMany — jinak nelze atributy vztahu uložit.
Shrnutí:
  • 1:1 → FK + UNIQUE v jedné tabulce (nebo sloučit).
  • 1:N → FK na straně "mnoha".
  • M:N → vazební tabulka (může mít vlastní atributy).

17. Typy uložišť (relační DB, NoSQL) – výhody a nevýhody

Ne všechna data se hodí do relační tabulky — NoSQL databáze řeší scénáře, kde relační model naráží na svá omezení: flexibilita schématu, horizontální škálovatelnost.

Relační databáze (RDBMS)

Příklady: PostgreSQL, MySQL, Oracle, SQL Server.
Výhody: ACID transakce, silná konzistence, SQL standard, komplexní dotazy s JOINy.
Nevýhody: Obtížná horizontální škálovatelnost (sharding), fixní schéma, výkon při masivní zápisové zátěži na distribuovaném clusteru.

NoSQL databáze – kategorie

TypPříkladyVhodné pro
DokumentovéMongoDB, CouchDBFlexibilní schéma, JSON dokumenty, CMS, katalogy
Klíč-hodnotaRedis, DynamoDBCache, session, jednoduché vyhledávání
SloupcovéCassandra, HBaseAnalytika, big data, časové řady
GrafovéNeo4j, NeptuneSociální sítě, doporučení, znalostní grafy
CAP teorém (Brewer, 2000): Distribuovaný systém nemůže současně garantovat všechny tři vlastnosti — může garantovat jen dvě ze tří:
  • Consistency: všechny uzly vidí stejná data ve stejný čas.
  • Availability: každý dotaz dostane odpověď (ne nutně aktuální).
  • Partition tolerance: systém funguje i při výpadku části sítě.
NoSQL obvykle volí AP (dostupnost + partition tolerance) → eventual consistency — data se nakonec shodnou, ale ne okamžitě.
Shrnutí:
  • RDBMS: ACID, SQL, silná konzistence; NoSQL: škálovatelnost, flexibilní schéma, eventual consistency.
  • Typy NoSQL: dokumentové, klíč-hodnota, sloupcové, grafové.
  • CAP teorém: v distribuovaném systému nelze mít C + A + P zároveň.

18. DML – kategorie příkazů, příklady

DML (Data Manipulation Language) je podmnožina SQL pro práci s daty — čtení, vkládání, aktualizace a mazání.

Příkazy DML:
  • SELECT — čtení dat (někdy vyčleňován do DQL)
  • INSERT — vkládání nových řádků
  • UPDATE — úprava existujících řádků
  • DELETE — mazání řádků
  • MERGE — kombinace INSERT + UPDATE (upsert)
Příklady: 
-- INSERT
INSERT INTO produkty (nazev, cena) VALUES ('Notebook', 25000);

-- UPDATE s podmínkou
UPDATE produkty SET cena = cena * 0.9 WHERE cena > 10000;

-- DELETE
DELETE FROM produkty WHERE cena < 100;

-- MERGE (upsert)
MERGE INTO cile USING zdroj ON (cile.id = zdroj.id)
WHEN MATCHED THEN UPDATE SET cile.hodnota = zdroj.hodnota
WHEN NOT MATCHED THEN INSERT (id, hodnota) VALUES (zdroj.id, zdroj.hodnota);
DML vs DDL vs DCL: DML pracuje s daty (řádky), DDL definuje strukturu (tabulky), DCL řídí přístupová práva. DML příkazy jsou transakční (lze je rolovat zpět); DDL příkazy jsou v mnoha DB auto-commit.
Shrnutí:
  • DML = SELECT, INSERT, UPDATE, DELETE, MERGE.
  • DML operuje s řádky a je transakční (rollbackovatelný).
  • MERGE kombinuje INSERT a UPDATE do jedné operace (upsert pattern).

19. Strict 2PL (Two-Phase Locking)

Strict Two-Phase Locking je protokol zamykání, který garantuje serializovatelnost a zároveň předchází kaskádovým rollbackům — jde o nejpoužívanější variantu 2PL v praxi.

2PL (Two-Phase Locking): Transakce prochází dvěma fázemi:
  1. Rostoucí fáze (Growing): Transakce smí pouze získávat zámky, nesmí žádný uvolnit.
  2. Klesající fáze (Shrinking): Transakce smí pouze uvolňovat zámky, nesmí žádný nový získat.
Tato podmínka garantuje acyklický precedenční graf → serializovatelnost.
Strict 2PL: Varianta 2PL, kde transakce drží všechny exkluzivní zámky (X-locky) až do COMMIT nebo ROLLBACK. Zabraňuje:
  • Kaskádovým rollbackům — jiná transakce nemůže přečíst nepotvrzená data.
  • Dirty read.
Nevýhoda: zámky jsou drženy déle → více blokování → nižší paralelismus.
Rozdíl 2PL vs Strict 2PL:
2PL: T1 může uvolnit X-lock na A uprostřed transakce, T2 může A přečíst — pokud T1 pak rollbackuje, T2 pracovala se špinavými daty (kaskádový rollback T2).
Strict 2PL: X-locky se drží do commitu → T2 musí čekat, ale data jsou vždy čistá.
Shrnutí:
  • 2PL: fáze získávání + fáze uvolňování → garantuje serializovatelnost.
  • Strict 2PL: X-locky až do commitu → zabraňuje dirty read a kaskádovým rollbackům.
  • Nevýhoda: vyšší počet deadlocků, nižší paralelismus.

20. Deadlock – Coffmanovy podmínky, detekce, prevence, graf čekání

Deadlock (uváznutí) nastane, když dvě nebo více transakcí čekají každá na zdroj, který drží ta druhá — bez vnějšího zásahu by čekaly věčně.

Coffmanovy podmínky (1971): Deadlock nastane právě tehdy, platí-li všechny čtyři podmínky současně:
  1. Vzájemné vyloučení (Mutual Exclusion): Zdroj může v daný moment používat jen jedna transakce.
  2. Drž a čekej (Hold and Wait): Transakce drží zdroje a přitom čeká na další.
  3. Bez preempce (No Preemption): Zdroje nelze násilně odebrat — transakce je musí sama uvolnit.
  4. Cyklické čekání (Circular Wait): Existuje cyklus T1 čeká na T2, T2 čeká na T3, ..., Tn čeká na T1.

Graf čekání (Wait-For Graph)

Uzly = transakce. Hrana $T_i \to T_j$ = $T_i$ čeká na zdroj, který drží $T_j$.
Deadlock ↔ cyklus ve wait-for grafu.

Detekce

Většina RDBMS (PostgreSQL, MySQL, Oracle) periodicky spouští algoritmus detekce cyklů ve wait-for grafu. Při nalezení cyklu je zvolena oběť (victim) — jedna transakce je ukončena (rollback), cyklus se přeruší a ostatní mohou pokračovat.

Prevence

  • Timeout: Transakce čeká max. N sekund, pak se odroluje.
  • Seřazené zamykání: Všechny transakce zamykají zdroje ve stejném pořadí → cyklus nemůže vzniknout (porušuje podmínku cyklického čekání).
  • Wait-Die / Wound-Wait: Priority podle věku transakce — starší čeká nebo zabije mladší, aby se zabránilo cyklu.
Příklad deadlocku:
T1: LOCK(A) → čeká na LOCK(B)
T2: LOCK(B) → čeká na LOCK(A)
Wait-for graf: T1 → T2 → T1 (cyklus) → deadlock.
Řešení: DB zvolí oběť (např. T2), provede rollback, T1 získá B a dokončí se.
Shrnutí:
  • Deadlock = cyklické čekání na zámky; nutné jsou všechny 4 Coffmanovy podmínky.
  • Detekce: cyklus ve wait-for grafu → rollback oběti.
  • Prevence: timeout, seřazené zamykání, age-based priority (Wait-Die, Wound-Wait).

21. Primární klíč vs. klíč, složené klíče

Správné rozlišení mezi kandidátním a primárním klíčem je základem normalizace a návrhu schématu.

Kandidátní klíč (Candidate Key): Minimální množina atributů, která jednoznačně identifikuje každý řádek relace. Relace může mít více kandidátních klíčů.
Primární klíč (Primary Key): Jeden vybraný kandidátní klíč — "oficiální" identifikátor. Automaticky dostává NOT NULL + UNIQUE constraint a je základem pro cizí klíče jiných tabulek.
Alternativní klíč (Alternate Key): Kandidátní klíče, které nebyly zvoleny jako primární. Implementovány jako UNIQUE NOT NULL.
Složený klíč (Composite Key): Klíč tvořený více atributy — žádný z nich sám nestačí k identifikaci, až jejich kombinace. Typický pro vazební tabulky.
Příklad: Tabulka Student(rc, email, jmeno).
Kandidátní klíče: {rc}, {email}.
Primární klíč: zvolíme {rc}.
Alternativní klíč: {email} → implementujeme jako UNIQUE NOT NULL.
Vazební tabulka Zapis(student_id, predmet_id): složený PK (student_id, predmet_id).
Surrogátní vs. přirozené klíče: Surrogátní klíče (umělé ID generované DB, např. SERIAL, UUID) jsou stabilnější — přirozené klíče (rodné číslo, email) se mohou "změnit". Surrogátní klíče nemají sémantiku; přirozené ano. Volba závisí na kontextu a stabilitě hodnot.
Shrnutí:
  • Kandidátní klíč: minimální unikátní identifikátor; může jich být více.
  • Primární klíč: jeden zvolený kandidátní klíč (NOT NULL + UNIQUE).
  • Složený klíč: více atributů dohromady tvoří identifikátor.

22. Pohledy (Views) – k čemu slouží

View (pohled) je uložený SELECT dotaz, ke kterému lze přistupovat jako k tabulce. Zjednodušuje práci s daty, aniž by kopíroval data samotná.

Pohled (View): Virtuální tabulka definovaná SELECT dotazem. Data fyzicky neexistují v pohledu — jsou načítána dynamicky z podkladových tabulek při každém přístupu.

K čemu pohledy slouží

  • Abstrakce a zjednodušení: Skryjeme složité JOINy za jednoduchou "tabulku".
  • Bezpečnost: Uživatel vidí jen povolené sloupce/řádky bez přístupu k základní tabulce.
  • Zpětná kompatibilita: Při změně struktury tabulky pohled zachová starý interface pro existující dotazy.
  • Opakované použití: Složitý dotaz definujeme jednou a používáme vícekrát.
Příklad: 
-- Vytvoření pohledu
CREATE VIEW aktivni_studenti AS
SELECT id, jmeno, email
FROM studenti
WHERE aktivni = TRUE;

-- Použití jako tabulka
SELECT * FROM aktivni_studenti WHERE jmeno LIKE 'Jan%';
Materializovaný pohled (Materialized View): Výsledek dotazu je fyzicky uložen a aktualizován periodicky nebo explicitně (REFRESH MATERIALIZED VIEW). Rychlejší čtení, ale data mohou být zastaralá. Vhodné pro analytické dotazy.
Upozornění: Standardní pohledy jsou obvykle read-only. INSERT/UPDATE přes pohled je možný jen u jednoduchých pohledů bez GROUP BY, DISTINCT nebo JOINů (závisí na DBMS).
Shrnutí:
  • View = uložený SELECT; neuchovává data, jen definici dotazu.
  • Použití: abstrakce, bezpečnost, zpětná kompatibilita, zjednodušení.
  • Materializovaný view ukládá fyzicky výsledek pro rychlejší čtení (stojí za refresh overhead).

23. Indexy v DB, B-stromy – operace, vizualizace

Index je datová struktura, která dramaticky zrychluje vyhledávání v tabulce — jako rejstřík v knize. Nejpoužívanější implementací je B+-strom.

Index: Pomocná datová struktura udržovaná DB vedle tabulky. Umožňuje rychlé vyhledání řádků bez prohledávání celé tabulky (full table scan). Cena: overhead při INSERT/UPDATE/DELETE (nutnost aktualizace indexu).

B+-strom

B+-strom (B+Tree): Vyvážený prohledávací strom pro databázové indexy. Vlastnosti:
  • Všechny datové záznamy jsou jen v listech (vnitřní uzly nesou jen navigační klíče).
  • Listy jsou navzájem propojeny (linked list) → efektivní sekvenční / rozsahové skenování.
  • Strom je vždy vyvážený — všechny listy jsou na stejné hloubce.
  • Složitost hledání, vkládání, mazání: $O(\log n)$.

Operace

OperacePostup
HledáníOd kořene porovnávej klíče, jdi do správného podstromu, opakuj do listu. $O(\log n)$.
VkládáníNajdi správný list, vlož. Je-li plný → split (rozděl, prostřední klíč jde do rodiče). Split se propaguje až do kořene.
MazáníNajdi a smaž klíč v listu. Je-li uzel pod minimálním počtem klíčů → merge se sourozencem nebo rotace klíče.
Vizualizace B+-stromu (řád 2, max 3 klíče na uzel): 
              [30 | 60]
             /    |    \
        [10|20] [40|50] [70|80|90]
           ↕       ↕        ↕
        listy propojeny pro range scan:
        10 → 20 → 40 → 50 → 70 → 80 → 90
Dotaz WHERE age BETWEEN 20 AND 70: najdi 20 v listu, pak sekvenčně procházej listy do 70.
Kdy index nepomáhá: Nízká selektivita (boolean, pohlaví), dotaz vrací velký procent tabulky, malé tabulky (full scan je rychlejší). Indexy také zpomalují zápisy.
Shrnutí:
  • Index = pomocná struktura pro rychlé vyhledávání; $O(\log n)$ místo $O(n)$.
  • B+-strom: data jen v listech, listy propojeny → efektivní bodové i rozsahové dotazy.
  • Split při vkládání, merge/rotace při mazání — strom zůstane vyvážený.

24. OLTP databáze – klíčové požadavky

OLTP (Online Transaction Processing) je typ databázového provozu charakteristický pro provozní systémy — banky, e-shopy, rezervační systémy.

OLTP: Databázový provoz s mnoha krátkými, souběžnými transakcemi, které čtou a zapisují malé množství dat. Důraz na nízkou latenci, souběžnost a spolehlivost.

Klíčové požadavky

PožadavekPopis
Nízká latenceTransakce musí proběhnout v milisekundách.
Vysoká souběžnostTisíce transakcí za sekundu; efektivní zamykání (2PL, MVCC).
ACID transakceKaždá transakce musí být atomická, konzistentní, izolovaná, trvanlivá.
NormalizaceData normalizována (3NF) pro minimalizaci redundance a rychlé zápisy.
Dostupnost 24/7Replikace pro failover, žádná plánovaná odstávka.
Příklady OLTP systémů: Bankovní převody, objednávkové systémy, rezervace letenek, POS terminály v obchodech.
OLTP vs OLAP: OLAP (Online Analytical Processing) je opak — málo složitých analytických dotazů nad obrovskými daty (datové sklady, reporty). OLAP je denormalizovaný (hvězdicové schéma pro rychlé agregace), OLTP je normalizovaný (pro rychlé transakce).
Shrnutí:
  • OLTP: mnoho krátkých transakcí, nízká latence, vysoká souběžnost.
  • Klíčové: ACID, normalizace, indexy, dostupnost 24/7.
  • Opak je OLAP (analytika) — denormalizovaný, pro čtení velkých dat.

25. Replikace, vysoká dostupnost

Replikace znamená udržování kopií dat na více serverech — základní nástroj pro vysokou dostupnost, odolnost proti výpadkům a škálovatelnost čtení.

Replikace: Automatické kopírování dat z primárního (master/primary) serveru na jeden nebo více sekundárních (replica/slave) serverů v reálném čase nebo se zpožděním.

Typy replikace

TypPopisTrade-off
SynchronníTransakce potvrzena až po zápisu na primary i repliceSilná konzistence; vyšší latence zápisů
AsynchronníPrimary potvrdí ihned, replika se aktualizuje se zpožděnímNižší latence; riziko ztráty dat při výpadku primary
Failover: Automatické přepnutí na záložní server při výpadku primárního. S replikací lze failover provést v sekundách (PostgreSQL Patroni, MySQL Group Replication).
Typická HA architektura:
Primary DB → 1 synchronní replika (failover kandidát) + N asynchronních replik (pro čtecí zátěž).
Load balancer distribuuje SELECT dotazy na repliky; INSERT/UPDATE/DELETE vždy na primary.
Replikace ≠ záloha! Replikace chrání proti výpadku serveru, ne proti datové chybě — smazání tabulky se okamžitě replikuje na všechny repliky. Záloha (dump, point-in-time recovery) je nutná zvlášť.
Shrnutí:
  • Replikace = kopie dat na více serverech; synchronní (konzistence) vs. asynchronní (nižší latence).
  • HA architektura: primary + repliky + automatický failover.
  • Repliky škálují čtení; zápisy vždy přes primary. Replikace není záloha.

26. Útoky na databáze (SQL injection)

SQL injection je jeden z nejzávažnějších bezpečnostních útoků — útočník "vpašuje" vlastní SQL kód do dotazu aplikace a získá neoprávněný přístup k datům nebo je poškodí.

SQL Injection: Útok, při němž útočník vloží škodlivý SQL kód do vstupu aplikace, který je následně vykonán databází. Vzniká, pokud aplikace přímou concatenací vkládá uživatelský vstup do SQL řetězce bez sanitizace.
Příklad zranitelného kódu: 
// ŠPATNĚ — přímá concatenace
String query = "SELECT * FROM users WHERE name='" + username + "'";

// Útočník zadá: ' OR '1'='1
// Výsledný dotaz: SELECT * FROM users WHERE name='' OR '1'='1'
// Vrátí všechny uživatele!

// Nebezpečnější vstup: '; DROP TABLE users; --
// Dotaz: SELECT * FROM users WHERE name=''; DROP TABLE users; --'
// Smaže celou tabulku users!

Obrana

1. Parametrizované dotazy (Prepared Statements) — nejúčinnější obrana: 
// SPRÁVNĚ — vstup je vždy data, nikdy kód
PreparedStatement ps = con.prepareStatement(
    "SELECT * FROM users WHERE name = ?");
ps.setString(1, username);
Parametr je vždy zpracován jako datový řetězec, nikdy jako součást SQL příkazu.
2. ORM: Hibernate, JPA generují parametrizované dotazy automaticky — implicitní ochrana.
3. Minimální oprávnění (Principle of Least Privilege): DB uživatel aplikace by měl mít jen nutná práva (SELECT, INSERT) — žádný DROP, žádný TRUNCATE, žádný CREATE.
4. Validace vstupu: Whitelist povolených hodnot. Jako doplněk, nikoli jediná obrana.
Upozornění: Escapování vstupu nestačí — různé DB mají různé escape konvence a útočníci nacházejí obcházení. Parametrizované dotazy jsou jedinou spolehlivou obranou.
Shrnutí:
  • SQL injection = vložení SQL kódu do vstupu → DB ho vykoná jako legitimní příkaz.
  • Příčina: přímá concatenace uživatelského vstupu do SQL řetězce.
  • Obrana: parametrizované dotazy (prepared statements) + minimální DB oprávnění.
