1. XSS (Cross-site Scripting)

Co to je a proč to vzniklo

XSS vzniká, když webová aplikace zahrne do výstupu uživatelský vstup bez dostatečného ošetření. Prohlížeč pak obsah považuje za legitimní kód stránky a spustí ho. Útočník tak může ve jménu oběti ovládat stránku — číst cookies, přesměrovávat, odesílat formuláře.

Definice: XSS (Cross-site Scripting) je třída zranitelností, při níž útočník injektuje spustitelný skript do webové stránky zobrazené jiným uživatelem, čímž skript běží v kontextu (originu) napadené aplikace.

Varianty

Příklad – Reflected XSS:
Aplikace zobrazí hledaný výraz: Výsledky pro: <script>alert(document.cookie)</script>. Útočník pošle oběti odkaz s payloadem v URL. Skript se neuloží na server, ale odrazí se zpět v odpovědi. Oběť musí kliknout na škodlivý odkaz — proto je tento typ méně nebezpečný než stored XSS, ale stále velmi rozšířený.
Příklad – Stored (Persistent) XSS:
Útočník uloží payload do databáze (komentář, profil, fórum). Každý uživatel, který danou stránku zobrazí, skript spustí automaticky. Stačí jednou uložit, útok se opakuje pro všechny návštěvníky — mnohem nebezpečnější varianta.
Pozor: Existuje ještě třetí varianta — DOM-based XSS, kde payload nikdy neopustí prohlížeč; manipuluje přímo DOM prostřednictvím JavaScript kódu na straně klienta (čtení z location.hash apod.).

Obrana

  • Output encoding — HTML entity encoding každého uživatelského vstupu vkládaného do HTML (&, <, >, "). Nejdůležitější obrana.
  • Content Security Policy (CSP) — HTTP hlavička, která říká prohlížeči, odkud smí načítat skripty. Zabrání spuštění inline skriptů a skriptů z cizích domén.
  • HttpOnly flag na cookies — zabrání JavaScriptu číst session cookie.
  • Validace vstupu — whitelistový přístup tam, kde je to možné.
Intuice: XSS je v podstatě problém mixing dat a kódu. Prohlížeč nedokáže rozlišit, co je data a co kód — proto musíme data enkódovat, aby je prohlížeč vždy interpretoval jako text, ne jako instrukce.
Shrnutí:
  • XSS = injekce skriptu útočníka do stránky jiného uživatele
  • Reflected: payload v URL, nevytrvalý; Stored: uložen v DB, zasáhne všechny návštěvníky
  • Hlavní obrana: output encoding + CSP + HttpOnly cookies

2. MVC architektonický vzor

Motivace

Bez struktury web aplikace rychle degeneruje do "spaghetti kódu" — HTML promíšené s SQL a business logikou. MVC odděluje odpovědnosti, aby různé části aplikace mohly být vyvíjeny a testovány nezávisle.

Definice: MVC (Model–View–Controller) je architektonický vzor rozdělující aplikaci do tří komponent: Model (data a business logika), View (prezentace dat uživateli) a Controller (zpracování uživatelského vstupu a orchestrace).

Komponenty a jejich odpovědnosti

KomponentaOdpovědnostCo NESMÍ dělat
Model Reprezentuje data, business pravidla, validace, přístup k DB. Zcela nezávislý na UI. Generovat HTML, pracovat s HTTP requestem
View Renderuje data z Modelu do HTML (nebo JSON). Obsahuje pouze prezentační logiku. Přistupovat přímo k DB, rozhodovat o business pravidlech
Controller Přijímá HTTP požadavky, volá Model, vybírá View a posílá data do View. Obsahovat business logiku nebo SQL dotazy

Směry interakce

Typický tok požadavku:
HTTP Request → Controller → Model (data) → Controller → View → HTTP Response
  1. Uživatel odešle formulář (POST /products)
  2. Controller přijme požadavek, validuje vstup
  3. Controller zavolá Product::create($data) na Modelu
  4. Model uloží data do DB
  5. Controller vybere View (nebo přesměruje)
  6. View obdrží data a renderuje HTML
Intuice: Controller je "dirigent" — sám nic nehraje, ale říká Modelu, aby přinesl data, a View, aby je zobrazil. View je čtenář, Model je knihovna. Tato separace umožňuje totéž View použít pro API (JSON) i pro HTML, nebo vyměnit databázi bez změny Views.
Shrnutí:
  • Model = data + logika, View = prezentace, Controller = orchestrace
  • Tok: Request → Controller → Model → Controller → View → Response
  • Benefity: testovatelnost, oddělení zodpovědností, znovupoužitelnost
  • Používají ho všechny moderní PHP frameworky (Laravel, Symfony)

3. SQL Injection

Princip útoku

SQL injection vzniká, když aplikace sestavuje SQL dotaz přímou konkatenací uživatelského vstupu. Útočník do vstupu vloží SQL syntaxi, která změní strukturu dotazu — místo dat interpreter dostane příkazy.

Příklad útoku na přihlašovací formulář:
Kód: SELECT * FROM users WHERE username='$user' AND password='$pass'
Útočník zadá: username = ' OR '1'='1' --
Výsledný dotaz: SELECT * FROM users WHERE username='' OR '1'='1' --' AND password='...'
Podmínka je vždy pravdivá, -- zakomentuje zbytek. Útočník se přihlásí bez hesla.
Pozor: SQL injection může vést nejen ke čtení dat, ale i k jejich modifikaci (DROP TABLE), ke čtení souborů, nebo při určitých konfiguracích i ke spuštění OS příkazů (xp_cmdshell v MS SQL).

Obrana

Parametrizované dotazy (Prepared Statements): Způsob sestavení SQL dotazu, kde struktura dotazu je oddělena od dat. Server nejprve zkompiluje šablonu dotazu, teprve pak dosadí data jako hodnoty — nikdy jako kód.
PHP PDO – správně: 
$stmt = $pdo->prepare('SELECT * FROM users WHERE username = ? AND password = ?');
$stmt->execute([$username, $password]);
Bez ohledu na to, co uživatel zadá, jeho vstup bude vždy interpretován jako hodnota, ne jako součást SQL.
  • ORM frameworky — Eloquent, Doctrine automaticky používají prepared statements
  • Principle of least privilege — DB uživatel aplikace nemá právo DROP TABLE
  • Validace vstupu — whitelisting přijatelných hodnot kde to dává smysl
Intuice: Prepared statements fungují proto, že databáze dostane plán dotazu před tím, než vůbec vidí data. Útočníkův vstup přichází až v druhém kroku a databáze ho nikdy nezpracuje jako instrukce — je to jako šuplík, do kterého lze dát cokoliv, ale jeho obsah se nikdy nespustí.
Shrnutí:
  • SQLi = útočník mění strukturu SQL dotazu přes uživatelský vstup
  • Hlavní obrana: prepared statements / parametrizované dotazy
  • Sekundárně: least privilege, ORM, validace vstupu

4. Bezpečnost webových aplikací

Přehled hlavních problémů (OWASP Top 10)

OWASP (Open Web Application Security Project) udržuje žebříček nejrozšířenějších zranitelností. Je de facto standardem pro bezpečnostní audit webových aplikací.

ZranitelnostPopisObrana
Broken Access Control Uživatel přistupuje k datům/funkcím, na které nemá právo (např. mění id v URL) Autorizační kontroly na serveru, RBAC
Injection (SQLi, XSS) Viz témata 1 a 3 Prepared statements, output encoding
Cryptographic Failures Citlivá data přenášena/uložena bez šifrování, slabé algoritmy (MD5, SHA1) HTTPS, bcrypt/Argon2 pro hesla, silné šifry
Insecure Design Bezpečnost nebyla zvažována při návrhu Threat modeling, secure SDLC
Security Misconfiguration Výchozí hesla, debug mód v produkci, zbytečné služby Hardening, pravidelné audity
CSRF Falzifikace požadavků — útočník přiměje prohlížeč oběti odeslat požadavek CSRF tokeny, SameSite cookies
Vulnerable Components Zastaralé knihovny se známými CVE Dependency scanning, pravidelné aktualizace

Klíčové principy obrany

  • Defense in depth — více vrstev obrany; prolomení jedné neznamená kompromitaci systému
  • Least privilege — každá komponenta má minimální potřebná práva
  • Fail securely — při chybě systém selže do bezpečného stavu, neodhaluje detaily
  • Validate at boundaries — validace vstupu na hranici systému (HTTP request), důvěra internímu kódu
Intuice: Většina zranitelností pramení z jediné chyby — aplikace důvěřuje uživatelskému vstupu. Zlaté pravidlo: veškerý vstup je nepřátelský, dokud není prokázán opak.
Shrnutí:
  • OWASP Top 10 je standardní přehled nejnebezpečnějších webových zranitelností
  • Nejrozšířenější: broken access control, injection, cryptographic failures
  • Klíčové principy: defense in depth, least privilege, validace vstupu

5. Architektura webových aplikací

Třívrstvá architektura

Moderní webové aplikace jsou typicky organizovány do tří logických vrstev, které mohou běžet na jednom nebo více fyzických serverech.

VrstvaOdpovědnostTechnologie
Prezentační vrstva
(Presentation / Client tier)		 Zobrazení UI, interakce s uživatelem, odeslání HTTP požadavků HTML, CSS, JavaScript, prohlížeč
Aplikační vrstva
(Business Logic / Middle tier)		 Business pravidla, zpracování požadavků, validace, orchestrace dat PHP, Python, Java, Node.js, .NET
Datová vrstva
(Data / Persistence tier)		 Trvalé uložení dat, dotazy, integrita dat MySQL, PostgreSQL, MongoDB, Redis

Proč oddělovat vrstvy?

  • Škálovatelnost — každá vrstva se škáluje nezávisle (přidat DB repliky bez změny app serveru)
  • Bezpečnost — databáze není přímo dostupná z internetu, ochráněna za aplikační vrstvou
  • Udržovatelnost — změna UI neovlivní business logiku
  • Testovatelnost — business logika testovatelná bez UI

Varianty nasazení

Příklady reálného nasazení:
  • Monolith na jednom serveru — vše na jednom stroji (vývoj, malé projekty)
  • Oddělená DB — aplikační a prezentační vrstva společně, databáze na dedikovaném serveru
  • Plně distribuované — každá vrstva na vlastní infrastruktuře, load balancery, CDN
  • Microservices / SoA — aplikační vrstva rozdrobena do nezávislých služeb (viz téma 11)
Shrnutí:
  • Tři vrstvy: prezentační (UI), aplikační (logika), datová (persistence)
  • Oddělení umožňuje nezávislé škálování, testování a nasazení
  • Databáze nikdy přímo přístupná z internetu

6. HTTP protokol

Základní charakteristiky

Definice: HTTP (Hypertext Transfer Protocol) je aplikační protokol fungující na principu požadavek–odpověď (request–response) nad TCP/IP. Je bezstavový (stateless) — každý požadavek je nezávislý, server si nepamatuje předchozí.

HTTP metody

MetodaÚčelIdempotentníBezpečná
GETNačtení zdroje, data v URLAnoAno
POSTVytvoření nového zdroje / odeslání datNeNe
PUTNahrazení celého zdrojeAnoNe
PATCHČástečná aktualizace zdrojeNe*Ne
DELETESmazání zdrojeAnoNe
HEADJako GET, ale bez těla odpovědiAnoAno
OPTIONSZjistit, jaké metody server podporujeAnoAno

* Bezpečná = nemění stav serveru. Idempotentní = opakované volání má stejný výsledek.

Struktura požadavku a odpovědi

HTTP Request: 
POST /api/login HTTP/1.1
Host: example.com
Content-Type: application/json
Authorization: Bearer eyJhbGciO...

{"username": "adam", "password": "secret"}
HTTP Response: 
HTTP/1.1 200 OK
Content-Type: application/json
Set-Cookie: session=abc123; HttpOnly; Secure

{"token": "eyJhbGciO..."}

Autorizace v HTTP

  • Basic Auth — Authorization: Basic base64(username:password). Jednoduché, ale heslo jde v každém požadavku (nutně HTTPS).
  • Bearer Token (OAuth 2.0 / JWT) — Authorization: Bearer <token>. Klient dostane token po přihlášení a posílá ho v každém požadavku. Token obsahuje claims (kdo je uživatel), server ověří podpis.
  • Session cookie — server vytvoří session, ID uloží do cookie; prohlížeč ji posílá automaticky.

Status kódy

SkupinaPříklady
2xx Úspěch200 OK, 201 Created, 204 No Content
3xx Přesměrování301 Moved Permanently, 304 Not Modified
4xx Chyba klienta400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found
5xx Chyba serveru500 Internal Server Error, 503 Service Unavailable
Intuice: HTTP je bezstavový záměrně — zjednodušuje škálování. Každý server zvládne zpracovat jakýkoli požadavek bez sdílení stavu. Stavovost (přihlášení, košík) je pak řešena explicitně na aplikační úrovni pomocí cookies nebo tokenů.
Shrnutí:
  • HTTP je bezstavový request-response protokol nad TCP/IP
  • Klíčové metody: GET (čtení), POST (vytvoření), PUT/PATCH (aktualizace), DELETE
  • Autorizace: Basic Auth, Bearer Token (JWT), session cookie
  • Status kódy: 2xx úspěch, 3xx redirect, 4xx chyba klienta, 5xx chyba serveru

7. Stavovost webové aplikace

Problém

HTTP je bezstavový — po každém požadavku server zapomene, s kým mluvil. Přitom aplikace potřebuje pamatovat: kdo je přihlášen, co je v košíku, jaký je jazyk. Stavovost musíme implementovat explicitně.

Klientská strana

Cookie: Malý textový soubor uložený v prohlížeči. Prohlížeč ho automaticky přikládá ke každému HTTP požadavku na příslušnou doménu. Nastavuje se hlavičkou Set-Cookie v odpovědi.
Atributy cookie:
  • HttpOnly — JavaScript nesmí číst cookie (ochrana před XSS krádeží)
  • Secure — cookie se posílá pouze přes HTTPS
  • SameSite=Strict/Lax — ochrana před CSRF (cookie se neposílá při cross-site requestech)
  • Expires / Max-Age — platnost cookie

Další klientská úložiště: localStorage (trvalé, přístupné JS, vhodné pro UI preference), sessionStorage (jen pro danou záložku), IndexedDB (strukturovaná data, offline).

Serverová strana

Session: Stav uložený na serveru, identifikovaný unikátním Session ID. Klient dostane jen ID (typicky v cookie), citlivá data zůstávají na serveru.
Průběh session:
  1. Klient pošle přihlašovací formulář
  2. Server ověří credentials, vytvoří session záznam v DB/paměti
  3. Server pošle Set-Cookie: session_id=abc123
  4. Každý další požadavek klient přikládá cookie s ID
  5. Server načte session data podle ID a ví, kdo je přihlášen

Srovnání

KritériumCookie (data u klienta)Session (data na serveru)
BezpečnostNižší — data viditelná/modifikovatelnáVyšší — data pod kontrolou serveru
Kapacita~4 KBNeomezená
ŠkálovatelnostBez problémůVyžaduje sdílené úložiště (Redis) při více serverech
VýkonData jdou v každém požadavkuJen ID jde v požadavku, data se načítají
Intuice: JWT tokeny jsou moderní alternativa session — celý stav je zakódován v tokenu podepsaném serverem. Token nese data (claims), server nemusí nic ukládat. Ideální pro API a microservices, kde sdílená session storage by byla zátěží.
Shrnutí:
  • HTTP je bezstavový — stavovost musíme vyřešit explicitně
  • Cookie: data u klienta (~4KB), server ji nečte, prohlížeč ji posílá automaticky
  • Session: data na serveru, klient má jen ID; bezpečnější, ale škálování vyžaduje sdílené úložiště
  • JWT: alternativa session pro API — stav zakódován v tokenu, server neukládá

8. Front controller vs. Page controller

Page Controller

Page Controller: Každá stránka/endpoint má vlastní PHP soubor, který zpracovává požadavek pro danou URL. Více vstupních bodů aplikace.
Struktura Page Controller: 
/products.php     → zpracovává /products
/product.php?id=1 → zpracovává /product/1
/cart.php         → zpracovává /cart
/admin/users.php  → zpracovává /admin/users
Každý soubor sám ověřuje přihlášení, načítá DB, renderuje šablonu. Logika se duplikuje.

Front Controller

Front Controller: Jediný vstupní bod (index.php) přijímá všechny HTTP požadavky. Centrální router mapuje URL na příslušný handler (controller třídu a metodu).
Tok požadavku ve Front Controller: 
Každý požadavek → index.php
                → Router::resolve($method, $path)
                → ProductController::show($id)
                → Response
URL rewriting v .htaccess zajistí, že všechny požadavky jdou přes index.php: 
RewriteEngine On
RewriteCond %{REQUEST_FILENAME} !-f
RewriteRule ^(.*)$ index.php [QSA,L]

Srovnání

KritériumPage ControllerFront Controller
Vstupní bodyVíce (jeden soubor = jedna stránka)Jeden (index.php)
Centralizace logikyObtížná — duplikace kóduSnadná (auth, logging, CSRF v jednom místě)
BezpečnostKaždý soubor musí ošetřit přístupCentrální kontrola — těžší zapomenout
SložitostJednodušší pro začátečníkyVyžaduje pochopení routeru
Vhodné proMalé projekty, skriptyFrameworky (Laravel, Symfony), větší projekty
Intuice: Page Controller je jako recepce v každém patře budovy — každé patro si samo hlídá přístupy. Front Controller je jako hlavní recepce v přízemí — všichni musí projít přes ni, takže bezpečnost a logování jsou garantované pro každého návštěvníka.
Shrnutí:
  • Page Controller: každá URL má svůj soubor; jednoduché, ale duplikace kódu
  • Front Controller: jeden vstupní bod, centrální router; základ moderních PHP frameworků
  • Front Controller umožňuje centrální autentizaci, logování, CSRF ochranu bez duplikace

9. PHP – serverování obsahu, GET requesty, routing

Jak PHP zpracovává GET požadavek

Webový server (Apache, Nginx) přijme HTTP GET požadavek. Pokud URL odpovídá PHP souboru (nebo je nakonfigurován rewriting), předá požadavek PHP interpreteru. Ten spustí skript, výstup skriptu se stane tělem HTTP odpovědi.

Přístup k datům GET požadavku v PHP: 
// URL: /products?category=books&sort=price

$_GET['category']          // 'books'
$_GET['sort']              // 'price'
$_SERVER['REQUEST_METHOD'] // 'GET'
$_SERVER['REQUEST_URI']    // '/products?category=books&sort=price'
$_SERVER['HTTP_HOST']      // 'example.com'

Routing

Routing je mapování URL na PHP kód. Cíl je mít pěkné URL (/products/42 místo /product.php?id=42) a centralizovanou logiku zpracování.

Jednoduchá implementace routeru: 
// routes.php
$router->get('/products', 'ProductController@index');
$router->get('/products/{id}', 'ProductController@show');
$router->post('/products', 'ProductController@store');

// Router::resolve($method, $path) vrátí callable
// index.php
$path = parse_url($_SERVER['REQUEST_URI'], PHP_URL_PATH);
$method = $_SERVER['REQUEST_METHOD'];
$handler = $router->resolve($method, $path);
$handler($request);

Dynamické parametry v URL

Extrakce parametrů: Routa /products/{id}/reviews/{reviewId} matchuje URL /products/42/reviews/7 a extrahuje id=42, reviewId=7. Router typicky používá regex: 
$pattern = '#^/products/(\d+)/reviews/(\d+)$#';
preg_match($pattern, $path, $matches);
// $matches[1] = 42, $matches[2] = 7

RESTful konvence

MetodaURLAkce
GET/productsIndex — seznam produktů
GET/products/{id}Show — detail produktu
POST/productsStore — vytvoření produktu
PUT/PATCH/products/{id}Update — aktualizace
DELETE/products/{id}Destroy — smazání
Shrnutí:
  • PHP přijme GET request přes webový server, data z URL jsou v $_GET
  • Routing mapuje URL patterny na PHP handlery (controller metody)
  • URL rewriting (.htaccess) přesměruje vše na index.php (front controller)
  • RESTful konvence: URL identifikuje zdroj, HTTP metoda identifikuje akci

10. AJAX

Princip a motivace

AJAX (Asynchronous JavaScript and XML): Technika komunikace webového prohlížeče se serverem na pozadí, bez nutnosti přenačtení celé stránky. Přenášená data nemusí být XML — dnes typicky JSON.

Klasický web: každá interakce (kliknutí, odeslání formuláře) způsobí kompletní reload stránky. AJAX umožňuje aktualizovat jen část stránky — výsledek vyhledávání, komentáře, notifikace — bez přerušení práce uživatele.

Moderní AJAX pomocí Fetch API: 
// Asynchronní — JS pokračuje dál, nečeká
fetch('/api/products/42')
  .then(response => response.json())
  .then(data => {
    document.getElementById('product-name').textContent = data.name;
  })
  .catch(err => console.error('Chyba:', err));

// Nebo s async/await
async function loadProduct(id) {
  const response = await fetch(`/api/products/${id}`);
  const data = await response.json();
  return data;
}

Asynchronnost a event loop

JavaScript je jednovláknový — nemůže skutečně paralelizovat. Místo toho má event loop: HTTP požadavek se odešle, JS pokračuje dál, a když přijde odpověď, callback se zařadí do fronty a spustí při nejbližší příležitosti.

Race condition v AJAXu

Race condition: Situace, kdy výsledek závisí na pořadí dokončení asynchronních operací, které nelze zaručit.
Typický scénář:
  1. Uživatel napíše "p" do vyhledávacího pole → odesláme request R1 pro "p"
  2. Napíše "ph" → odesláme request R2 pro "ph"
  3. R2 přijde dříve — zobrazíme výsledky pro "ph" ✓
  4. R1 přijde se zpožděním — přepíšeme výsledky pro "p" ✗ (špatně!)

Řešení race condition

  • Zrušení předchozího požadavku — AbortController umožní přerušit fetch request před odesláním nového
  • Verziování — každý request dostane číslo, odpovědi s nižším číslem se ignorují
  • Debouncing — čekáme, až uživatel přestane psát (300ms), pak teprve odešleme
  • Centralizovaný stav — Redux/Flux pattern: všechny aktualizace jdou přes jeden dispatcher, který zaručuje konzistenci
AbortController řešení: 
let controller = null;

function search(query) {
  if (controller) controller.abort(); // zruší předchozí
  controller = new AbortController();

  fetch(`/api/search?q=${query}`, { signal: controller.signal })
    .then(r => r.json())
    .then(data => showResults(data))
    .catch(err => { if (err.name !== 'AbortError') console.error(err); });
}
Intuice: Asynchronní kód je jako objednávka v restauraci — objednáš, číšník odejde, ty mezitím pijete víno. Problém nastane, když objednáš dvakrát a nevíš, které jídlo přijde první. AbortController je ekvivalent "vlastně, zruš tu první objednávku".
Shrnutí:
  • AJAX = asynchronní HTTP požadavky na pozadí bez reload stránky
  • JS je jednovláknový, asynchronnost řeší event loop a callbacky/promises
  • Race condition: odpovědi přicházejí v jiném pořadí než byly odeslány
  • Řešení: AbortController, debouncing, verziování, centralizovaný stav

11. SoA (Service-Oriented Architecture) a stavovost, škálovatelnost

Co je SoA

SoA (Service-Oriented Architecture): Architektonický přístup, kde aplikace je sestavena z nezávislých služeb, které spolu komunikují přes dobře definovaná rozhraní (typicky HTTP/REST nebo SOAP). Každá služba zapouzdřuje jednu business doménu a může být nasazena a škálována nezávisle.

SoA vzniklo jako odpověď na problémy monolitických aplikací: monolith je těžké škálovat (musíš škálovat vše najednou), obtížně udržitelný (změna jedné části rozbije jinou) a technologicky rigidní (celý systém musí používat stejný jazyk/framework).

Klíčové principy

  • Loose coupling — služby jsou minimálně závislé na ostatních; komunikují přes rozhraní, ne přímé volání
  • High cohesion — každá služba dělá jednu věc dobře (objednávky, platby, notifikace)
  • Interoperabilita — služby mohou být v různých technologiích, komunikují přes standardní protokoly
  • Znovupoužitelnost — stejná služba může obsluhovat více klientů (web, mobilní app, interní nástroje)
Příklad: e-shop jako SoA
  • User Service — správa uživatelů, autentizace
  • Product Service — katalog produktů, zásoby
  • Order Service — vytváření a správa objednávek
  • Payment Service — zpracování plateb
  • Notification Service — emaily, push notifikace
Každá služba má vlastní databázi, nasazuje se nezávisle, škáluje se podle potřeby (Payment Service ve špičce potřebuje víc instancí).

Stavovost v SoA

Distribuované prostředí komplikuje správu stavu. Preferovaný přístup: bezstavové služby — každý požadavek nese veškeré potřebné informace, stav je externalizován.

PřístupPopisVýhoda
Bezstavové služby Stav uložen externě (Redis, DB). Každá instance může obsluhovat jakýkoli request. Snadné škálování
JWT tokeny Stav (identita uživatele) zakódován v tokenu podepsaném serverem. Bez sdíleného úložiště
Event sourcing Stav se rekonstruuje z posloupnosti událostí (event log jako single source of truth). Auditovatelnost, time-travel

Škálovatelnost v SoA

  • Nezávislé škálování — pokud Payment Service je přetížena, přidáme instance jen pro ni
  • Fault isolation — výpadek Notification Service nepadne celý e-shop
  • API Gateway — centrální vstupní bod, který routuje požadavky na správné služby, řeší autentizaci a rate limiting
  • Message queues — Kafka, RabbitMQ pro asynchronní komunikaci; Order Service vloží zprávu, Payment Service ji zpracuje podle kapacity

Problémy a omezení

Pozor: SoA přináší komplexitu distribuovaných systémů:
  • Síťová latence — místo lokálního volání jde požadavek přes síť
  • Eventual consistency — nelze garantovat silnou konzistenci dat napříč službami
  • Debugging — trasování požadavku přes 5 služeb je složité (nutné distributed tracing)
  • Operační overhead — desítky služeb = desítky deploymentů, monitoringů, logů
Intuice: SoA je jako rozdělit velkou firmu na menší divize — každá má svou zodpovědnost a může se rozrůstat nezávisle. Ale koordinace mezi divizemi je složitější než v malé firmě. Microservices jsou extrémní verze SoA — velmi jemné rozdělení, každý service je malý a jednoduchý. SoA je vhodné tehdy, kdy monolith začíná brzdit vývoj a různé části aplikace mají výrazně odlišné nároky na škálování.
Shrnutí:
  • SoA = aplikace jako sada nezávislých, komunikujících služeb; každá za svou doménu
  • Stavovost: preferovat bezstavové služby + externalizace stavu (Redis, JWT)
  • Škálování: každá služba nezávisle, fault isolation, API Gateway, message queues
  • Cena: síťová latence, eventual consistency, operační složitost
